# RAG 系统整体架构：从文档摄入到答案生成的完整链路

## 一、为什么需要从架构视角看 RAG

把 RAG 理解成“检索 + 生成”两个字拼接在一起，是入门者最常见的误区。一个能在生产环境稳定运行的 RAG 系统，本质上是一条由**离线索引流水线**与**在线检索链路**双轨驱动的数据管道：前者负责把异构知识沉淀为可被高效召回的结构化表示，后者负责把用户的一个模糊问题转化为有依据、可追溯、低延迟的回答。

两条链路在时间维度上完全解耦——离线链路以分钟到小时为节奏运行，在线链路以百毫秒为节奏响应请求——但它们共享同一套**语料表示契约**（相同的分块规则、相同的 Embedding 模型、相同的词典与归一化策略）。任何对契约的单方面修改，都会导致召回质量断崖式下降。因此架构师的第一要务，是清晰划定各组件的职责边界与协作接口，而不是急于堆砌模型。

## 二、整体架构总览

下图以 ASCII 形式刻画了从文档摄入到答案生成的端到端流水线，左侧为离线索引，右侧为在线检索，中间以向量库与倒排库作为共享存储衔接：

```
        ┌─────────────────── 离线摄入链路 ───────────────────┐
        │                                                    │
  原始文档 → 文档加载 → 解析/抽取 → 清洗 → 分块 → 向量化 ──┐
   (PDF/HTML/    │           │           │         │       │
    Office/DB)   │           │           │         │       │
        │        ▼           ▼           ▼         ▼       │
        │   元数据归一     表格/图     句法/语义   Embedding  │
        │                                                    │
        │   并行: BM25 分词 → 倒排索引  ←─────────────────────┤
        │                                                    │
        └──────────────────────────────────┬─────────────────┘
                                           │
                              ┌────────────▼────────────┐
                              │  共享存储层              │
                              │  • 向量库 (HNSW/IVF)    │
                              │  • BM25 倒排库           │
                              │  • 元数据库 (PostgreSQL) │
                              └────────────┬────────────┘
                                           │
        ┌─────────────────── 在线检索链路 ──────────────────┐
        │                                                  │
   用户查询 → 查询接收 → 查询分析/重写 → 混合检索 → 重排 → │
                          │              │           │       │
                     多查询/子问题    向量+BM25    Cross-Encoder│
                     路由/意图判定   并行召回     业务规则过滤   │
                          │                          │       │
                          └──────────────→ 上下文组装 ──────┤
                                            │              │
                                      引用/去重/截断         │
                                            │              │
                                         LLM 生成           │
                                            │              │
                                     引用标注 → 返回前端    │
        └──────────────────────────────────────────────────┘
```

整条链路的“心脏”是中间的共享存储层：离线侧不断往里写，在线侧不断从里读。两侧通过 **chunk id + 版本号** 维持一致性，而不是依赖运行时调用。

## 三、离线摄入链路详解

离线链路的核心目标是把“人读得懂的文档”转化为“机器召得准的索引”。它通常包含五个阶段，且每个阶段都可能成为最终质量的瓶颈点。

### 3.1 文档加载

负责对接异构数据源：本地文件系统、对象存储、Confluence/Wiki、数据库快照、爬虫结果等。此阶段的工程难点不在“读”，而在**增量同步与变更检测**。成熟的做法是为每份文档维护一个指纹（内容哈希 + 修改时间），调度器据此判断是否需要重新进入流水线，避免全量重建。

### 3.2 解析与抽取

不同格式需要不同解析器：PDF 走版面分析、HTML 走 DOM 清洗、Office 走结构化反序列化。此阶段的关键产出不只是纯文本，还包括**结构信号**——标题层级、段落归属、表格、图注、页码。这些信号会成为后续分块与引用的重要元数据。对于扫描件，则需要前置 OCR。

### 3.3 清洗与归一

去除页眉页脚、水印、乱码、重复段落；统一编码、空白与标点；合并被错误换行打断的句子。这一步常被低估，但它是决定后续分块质量的隐形地基。

### 3.4 文本分块

将长文档切分为可被向量化的语义单元。分块策略直接影响召回精度与上下文完整性——过小则语义碎片化，过大则稀释了相似度信号。常用策略包括固定长度滑动窗口、句法边界切分、递归层次切分、以及基于语义相似度的动态切分。块大小、重叠窗口、是否保留父子结构（parent-child chunking），是此阶段需要调优的核心参数。

### 3.5 向量化与双索引入库

对每个分块调用 Embedding 模型生成稠密向量，同时将其文本送入 BM25 分词器构建倒排索引。两条索引**共享同一个 chunk id**，以便在线检索时能把向量召回与词法召回的结果对齐合并。元数据（来源、标题、时间、权限标签）写入关系库，用于过滤与追溯。

整个离线链路应是**幂等且可重跑**的：以文档 id 为幂等键，重新跑应覆盖旧分块并更新版本号，而非产生重复数据。

## 四、在线检索链路详解

在线链路的核心目标是把用户的一个自然语言问题，转化为一个“证据充分、上下文紧凑、引用可溯”的提示词，交由 LLM 生成。

### 4.1 查询接收

入口服务负责接收请求、鉴权、会话管理、限流与链路追踪。此层不触碰业务逻辑，只做协议适配与可观测性埋点。

### 4.2 查询分析与重写

这是现代 RAG 区别于“朴素检索 + 生成”的分水岭。用户问题往往含糊、口语化、包含指代或时态歧义。查询分析层需完成：

- **意图判定与路由**：判断是闲聊、事实问答、多步推理，还是需要调用工具，据此决定走轻量检索还是深度检索。
- **查询重写**：补全指代、消除歧义、把口语转为检索友好的表述。
- **多查询/子问题分解**：把复合问题拆为若干可独立检索的子查询，分别召回后合并。
- **查询扩展**：加入同义词、缩写展开，提升召回覆盖。

### 4.3 混合检索

并行执行两路召回：向量检索基于语义相似度捕捉“意思相近”的块，BM25 检索基于词项匹配捕捉“字面命中”的块。两路结果通过 RRF（Reciprocal Rank Fusion）或加权分数融合。混合检索同时解决了“同义不同字”与“同字不同义”两类经典难题。

### 4.4 重排

粗排召回的 Top-K（通常 50–100）需要用更重的 Cross-Encoder 模型做精排，把真正与查询相关的少数块排到前面。重排是性价比极高的质量提升手段——代价是百毫秒级延迟，换来的是上下文信噪比的显著跃升。

### 4.5 上下文组装

将重排后的 Top-N（通常 5–10）块按相关性排序拼接，做去重、引用编号、长度截断（受 LLM 上下文窗口约束），并注入系统提示与对话历史。组装质量直接决定 LLM 会不会“看着证据瞎编”。

### 4.6 LLM 生成与引用

LLM 基于组装好的上下文生成回答，并要求在关键论断后标注引用编号。生成后可加一层校验：引用是否真实存在于上下文中，避免幻觉式引用。

## 五、离线与在线链路对比

| 维度 | 离线摄入链路 | 在线检索链路 |
|------|--------------|--------------|
| 触发方式 | 定时/事件驱动 | 用户请求驱动 |
| 延迟要求 | 秒~分钟级，可容忍 | 百毫秒级，严格 |
| 吞吐目标 | 高吞吐批处理 | 高并发低延迟 |
| 主要产物 | 向量库 + 倒排库 + 元数据 | 引用标注的答案 |
| 失败影响 | 索引滞后，召回缺失 | 直接答错或超时 |
| 一致性要求 | 幂等、版本化 | 读最新版本即可 |
| 扩展瓶颈 | Embedding 算力、解析 I/O | 向量库 QPS、LLM 并发 |

## 六、各组件职责与边界

清晰划定边界，是避免“大泥球”架构的前提：

- **文档加载器**：只负责取数据，不解析。
- **解析器**：只负责把格式转为结构化文本，不切分。
- **分块器**：只负责切分与编号，不向量化。
- **向量化服务**：只负责 Embedding，不落库。
- **索引服务**：只负责写入向量库与倒排库。
- **查询分析层**：只负责理解与重写，不检索。
- **检索层**：只负责召回与融合，不生成。
- **重排层**：只负责精排，不修改召回集。
- **编排层**：串联以上各环节，处理超时、降级、缓存与可观测性，但**不持有业务状态**。

一个常见的反模式是把检索、重排、生成逻辑揉进同一个“智能体”里。短期看似简单，长期会导致难以测试、难以替换模型、难以做 A/B 实验。

## 七、模块化设计原则

1. **契约优先**：组件之间通过明确的 schema 通信（如统一的 `Chunk`、`Query`、`RetrievalResult` 结构），而非共享内存对象。
2. **无状态服务**：在线各组件应无状态，状态外置到缓存或存储，便于水平扩展。
3. **可替换性**：Embedding 模型、向量库、LLM 都应是可插拔接口，切换实现不应波及编排层。
4. **可观测性**：每个阶段产出埋点（耗时、召回数、相关性分数），支持全链路 trace。
5. **降级与熔断**：重排超时可降级为粗排结果，LLM 失败可回退缓存答案。

## 八、典型部署拓扑

生产级 RAG 通常按职责拆分为多个独立部署单元：

```
┌─────────┐    ┌─────────────┐    ┌──────────────────┐
│ 前端 UI │───▶│ 编排服务    │───▶│ 查询分析服务      │
└─────────┘    │ (Agent)     │    │ (Rewrite/Route)  │
               └──────┬──────┘    └────────┬─────────┘
                      │                    │
                      ▼                    ▼
               ┌──────────────────────────────┐
               │  存储与检索层                 │
               │  (解析/分块/索引/检索一体)    │
               └──────────────┬───────────────┘
                              │
            ┌─────────────────┼─────────────────┐
            ▼                 ▼                 ▼
       向量库            BM25 倒排库        元数据库
```

编排服务作为唯一对外入口，负责会话编排与结果聚合；查询分析服务独立部署以便单独扩容（重写模型较重）；存储层把解析、分块、索引、检索合为一体，既服务离线写入也服务在线读取，保证两侧使用同一套分块与归一逻辑。

## 九、本项目架构对照

本项目即采用上述四模块架构，落地为四个独立服务，端口分工清晰：

- **frontend**：负责用户交互、会话展示与引用渲染，对应“前端 UI”。
- **agent-service（编排，端口 28081）**：对应“编排服务”，作为唯一对外入口，串联查询分析、检索、上下文组装与 LLM 生成，处理超时降级与链路追踪，自身不持有业务状态。
- **query-rewrite-service（端口 28082）**：对应“查询分析服务”，承担意图判定、查询重写、子问题分解与路由决策，可独立扩容以应对重写模型的算力开销。
- **storage-layer（端口 28083）**：对应“存储与检索层”，把文档解析、分块、向量化、BM25 建库与混合检索合并为一处，确保离线写入与在线读取共享同一套分块规则与归一策略，避免契约漂移。

这一映射的关键价值在于：把通用 RAG 架构中“查询分析”这一最容易被忽视却最影响质量的环节，单独提升为一等公民服务；同时通过 storage-layer 的“读写同源”设计，从工程结构上杜绝了离线/在线分块不一致的隐患。三个端口（28081/28082/28083）之间通过明确的服务间契约通信，任一模块均可独立替换实现或水平扩展。

## 十、小结

RAG 系统的架构本质，是把“知识沉淀”与“知识调用”这两件节奏迥异的事，通过一套共享的表示契约稳定地衔接起来。离线链路追求的是**覆盖、精度、幂等**，在线链路追求的是**理解、召回、低延迟**，而编排层则负责让两者在工程上协作得既松耦合又高可靠。理解了离线与在线的分轨、各组件的边界、以及模块化设计的取舍，架构师就能在面对具体业务时，有依据地决定哪里该重投入、哪里该做降级，而不是把希望全部寄托在某一个更强模型上。
