# Agent 架构与 ReAct/Plan-Execute：推理-行动循环与工具调用

## 一、什么是 LLM Agent

LLM Agent（智能体）是以大语言模型为推理核心、能感知环境、自主决策并调用外部工具以完成复杂目标的系统。其本质是「以 LLM 为大脑，以工具为手脚，以循环为神经」的控制回路。一个工程化的 Agent 通常由四个要素构成：

- **规划（Planning）**：将用户目标分解为可执行步骤，必要时动态调整。
- **记忆（Memory）**：短期记忆承载当前任务上下文与中间状态，长期记忆承载跨会话经验。
- **工具（Tools）**：检索、计算、代码执行、API 调用、数据库查询等可被 LLM 调用的外部能力。
- **行动（Action）**：通过工具调用对环境施加影响，并将观测结果反馈回推理循环。

Agent 的关键特征不在于「能不能用工具」，而在于「能根据观测结果自主决定下一步做什么」，这正是 Agent 与单次推理的根本分野。

## 二、Agent 与单次 RAG 的区别

单次 RAG 是「检索-生成」的线性流水线：用户提问 → 检索 Top-K 片段 → 拼接 Prompt → 一次性生成回答。它没有循环、没有决策分支、对检索失败几乎没有自我修正能力。Agent 则完全不同：

| 维度 | 单次 RAG | LLM Agent |
|---|---|---|
| 决策方式 | 固定流水线 | 自主、动态决策 |
| 推理步数 | 单步生成 | 多步迭代 |
| 工具使用 | 仅检索 | 检索、计算、代码、API、子 Agent |
| 失败处理 | 几乎无 | 可观测、可重试、可改路 |
| 状态 | 无状态或仅上下文 | 短期/长期记忆 |
| 延迟 | 低（一次 LLM） | 高（多次 LLM + 多次工具） |
| 成本 | 可预测 | 随步数累积，难预测 |
| 可靠性 | 较高但能力受限 | 能力强但易发散、易幻觉 |

一句话概括：**RAG 是「查一次答一次」，Agent 是「想一步做一步、看到结果再想下一步」**。代价是更复杂的控制流与更高的不确定性，换来的是远超单次检索的任务求解能力。

## 三、ReAct 范式：推理与行动交替

ReAct（Reasoning + Acting）是 Yao 等人于 2022 年提出的经典 Agent 范式，核心思想是让 LLM 在「思考（Thought）」与「行动（Action）」之间交替推进，并以环境返回的「观测（Observation）」作为下一步推理的输入。这种显式的推理链路使 Agent 决策过程可解释、可调试，也显著降低幻觉与发散。

### ReAct 循环示意

```
用户问题 Q
     │
     ▼
 Thought #1: 分析问题，决定检索什么
 Action  #1: search(query="...")
 Observation #1: [检索返回的片段]
     │
 Thought #2: 片段不够，需要计算
 Action  #2: calculator(expr="...")
 Observation #2: [计算结果]
     │
 Thought #3: 信息已充分，可以作答
 Action  #3: finish(answer="...")
     ▼
   最终答案
```

每一轮迭代中，LLM 先产出一段「Thought」描述当前推理，再产出一个「Action」（通常是工具调用及其参数），运行时执行该工具并将「Observation」追加回上下文，循环往复直到触发终止动作（如 `finish`）或达到步数上限。

### ReAct 的工程要点

- **Prompt 结构化**：用少量示例演示 Thought/Action/Observation 格式，或使用模型原生 tool-calling 接口让 Action 以结构化方式产出。
- **上下文累积**：每轮 Thought 与 Observation 都追加进上下文，需控制总长度，必要时做摘要压缩。
- **错误即观测**：工具调用失败本身也是 Observation，Agent 应能基于错误信息调整策略而非直接崩溃。
- **优势**：实现直观、可解释性强、对开放任务适应性好；**劣势**：每步都调用一次 LLM，延迟与成本随步数线性增长，且缺全局规划易陷入局部最优。

## 四、Plan-and-Execute 范式：先规划后执行

Plan-and-Execute 针对 ReAct「走一步看一步」的局限，将 Agent 拆为两阶段：先用 Planner 生成完整步骤序列，再由 Executor 逐步执行，必要时回到 Planner 重新规划。典型流程：

1. **Planner** 接收用户目标，产出有序子任务列表（如：①检索资料；②提取关键数据；③计算汇总；④生成报告）。
2. **Executor** 依次执行每个子任务，每步内部可以是独立 ReAct 循环或单次工具调用。
3. **Re-planner**：当某步结果与预期偏差较大或出现新信息时，触发重新规划，更新剩余步骤。

与 ReAct 相比，Plan-Execute 的关键区别在于**全局视角**：动手前就对整条路径有判断，减少反复试探；但若初始规划质量不佳，后续执行会被带偏。工程上常采用「粗规划 + 执行中动态修正」的折中方案，并对长任务做层级化分解。

## 五、Reflexion 与自我反思

Reflexion（反思）范式在 ReAct 基础上引入「自我评估-反思-记忆」机制：Agent 在一次尝试结束（或失败）后，由自身或评估器生成一段反思，指出本次推理哪里出错、工具哪里用错，并将该反思写入记忆，作为下次尝试的指导。其循环可概括为：

```
执行尝试 → 评估（成功/失败 + 评分） → 反思（生成可改进经验） → 写入记忆 → 下一次尝试
```

Reflexion 不改变单步推理结构，而是通过**跨尝试的自我修正**提升成功率，特别适合代码生成、推理题等可验证任务。其局限在于反思本身也是 LLM 生成，可能引入新偏差，且需要可验证的评估信号才能形成有效反馈闭环。## 六、工具调用（Tool/Function Calling）机制

工具调用是 Agent 行动能力的落地方式。现代主流 LLM（Claude、GPT、Gemini 等）大多提供原生 function/tool calling 接口，其机制如下：

- **工具声明**：用 JSON Schema 描述每个工具的名称、用途、参数与返回结构，注入系统提示或 API 的 `tools` 字段。
- **模型决策**：LLM 在推理时决定是否调用工具，并以结构化 JSON 给出工具名与参数（而非自由文本）。
- **运行时执行**：宿主程序拦截该调用请求，实际执行工具（发 HTTP、跑代码），将结果以 `tool_result` 回填给模型。
- **循环往复**：模型基于工具结果继续推理，可能再次调用工具，直至给出最终答案。

相比早期 ReAct 用正则解析自由文本中的 Action，原生 tool calling 显著提升了参数准确性与可解析性。工程实践需注意：工具过多会稀释模型注意力，建议按任务做工具子集路由；工具描述要清晰、含失败语义；幂等工具可自动重试，副作用工具（写库、发消息）需谨慎处理重试与回滚；对工具结果做裁剪与结构化摘要，避免上下文膨胀。

## 七、Agent 循环控制

Agent 的核心是循环，而循环必须有可靠的控制机制，否则极易失控。关键控制点包括：

- **终止条件**：明确「什么算完成」，如模型主动调用 `finish`、给出 `end_turn` 信号或检测到不再产生新 Action。
- **最大步数（max_iterations）**：硬性上限防死循环，通常设 5–20 步，复杂任务可放宽但需配套监控。
- **超时与预算**：对总耗时与总 token 消耗设上限，超限即强制收尾。
- **错误恢复**：工具失败时区分可重试错误（网络抖动）与致命错误（参数非法），前者指数退避重试，后者回退到上一决策点或终止。
- **重复检测**：监测「连续相同 Action」或「相同参数反复调用」，触发去抖或强制改路，避免在死胡同里打转。
- **状态持久化**：对长任务做检查点，支持中断恢复与人工介入。

成熟框架通常把这些控制逻辑下沉到运行时（Runtime/Executor），让上层只关注 Prompt 与工具，避免业务代码与控制流耦合。

## 八、多 Agent 协作简述

当任务复杂到单一推理链难以胜任时，可采用多 Agent 协作架构。常见模式包括：

- **路由分发**：Supervisor Agent 根据意图将子任务分派给专业子 Agent。
- **流水线协作**：多个 Agent 串行接力，前者输出作为后者输入。
- **辩论/投票**：多 Agent 独立求解后交叉评审，提升鲁棒性。
- **层级化**：顶层 Manager 拆解任务，下层 Worker 执行，结果逐层聚合。

多 Agent 带来分工与并行的好处，也引入通信开销、状态同步、结果一致性、可观测性下降等难题。实践中应遵循「能单 Agent 不多 Agent」——仅当任务确实需要异质能力或并行加速时才拆分，并显式设计通信与汇总策略。

## 九、可靠性与成本/延迟挑战

Agent 能力强，但工程化的坑也多，主要体现在四个方面：

- **可靠性**：多步推理误差累积，单步幻觉在循环中会被放大；缺乏强验证信号时易「自信地错下去」。需引入校验、反思、置信度门控。
- **成本**：每步一次 LLM 调用，token 与算力成本随步数线性甚至超线性增长。需通过缓存、模型分层、步数压缩控制。
- **延迟**：串行 Thought-Action-Observation 链路使端到端延迟显著高于单次 RAG。可并行子任务应并行执行，长任务用流式输出改善体感。
- **安全与可观测**：Agent 自主调用工具可能触发副作用，必须有日志、审计、权限沙箱与人工确认机制，防止「自作主张」造成不可逆损害。

这些挑战决定了 Agent 不是「写个循环就上线」的系统，而是需要评估集、回归测试与运行时监控支撑的工程产物。## 十、小结与范式选型

综上，Agent 的本质是「以 LLM 为推理核心、以工具为行动能力、以循环为控制结构」的自主求解系统。ReAct、Plan-Execute、Reflexion 并非互斥，而是可组合的积木。选型建议：

- **步骤少、开放性强、需可解释**：首选 ReAct，实现简单、过程透明。
- **目标明确、路径较长、需全局规划**：选 Plan-Execute，先规划后执行，必要时动态重规划。
- **可验证任务、首轮易错**：叠加 Reflexion，用自我反思跨尝试修正。
- **单 Agent 能力受限、需异质专长或并行**：引入多 Agent 协作，并配 Supervisor 路由。
- **任何范式**：都务必做好循环控制（终止条件、最大步数、错误恢复、重复检测）与成本/延迟/安全治理。

工程上更常见的不是「纯范式」，而是**混合架构**：用 Plan-Execute 做骨架，每步内部用 ReAct，失败时用 Reflexion 反思，复杂子任务再派发子 Agent。理解每种范式的机理、权衡与适用边界，比执着于某一个框架更重要——这也是 Agent 架构师从「会用工具」走向「能设计可控、可演进系统」的关键。
