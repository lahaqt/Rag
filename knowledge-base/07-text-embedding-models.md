# 文本嵌入原理与模型选型：语义空间、维度与主流模型

在 RAG 系统中，检索质量的下限由分块策略决定，上限则几乎由嵌入模型决定。一个糟糕的嵌入会让最完美的向量索引也召回不到正确的文档；而一个合适的嵌入，则能让朴素的余弦相似度也能逼近语义检索的天花板。本文从嵌入的数学本质出发，讲清维度、归一化、相似度度量这些常被忽略的工程细节，再落到主流模型的横向对比与中文场景的选型决策，最后对照本项目 storage-layer 的嵌入客户端抽象。

## 一、嵌入的本质：把文本映射到高维语义空间

文本嵌入（Text Embedding）本质是一个函数 $f: \text{text} \to \mathbb{R}^d$，把任意长度的自然语言文本压缩成一个固定维度的稠密实数向量。这个向量并非随机的坐标，而是被训练成一种「语义坐标」：语义越接近的文本，向量在空间中的距离越近；语义越远的文本，距离越远。

理解这一点要抓住两个关键性质：

1. **语义连续性**。空间不是离散的格子，而是连续的流形。「猫」「狗」会比「猫」「汽车」更靠近，「苹果（水果）」和「苹果（公司）」则会落在中间地带——这正是多义词带来的固有歧义，也是嵌入检索需要面对的天然难题。
2. **结构可组合性**。训练良好的嵌入空间里，方向本身承载语义。经典的「国王 − 男人 + 女人 ≈ 女王」就是这种线性结构的体现。虽然这种性质被高估了，但它说明向量空间不是黑盒，而是有一定可解释结构的语义拓扑。

对 RAG 而言，嵌入的核心作用是把查询与文档「拉进同一个语义空间」比较距离。因此，**查询与文档必须使用同一个嵌入模型、同一种归一化方式**——这是不可妥协的铁律。一旦中途换模型，整个向量库就等同作废，必须重新索引。

## 二、对比学习：嵌入是怎么被训练出来的

现代嵌入模型几乎都基于**对比学习（Contrastive Learning）**训练。其核心思想是构造大量「正样本对」（语义相似的文本对）与「负样本对」（语义不相似的文本对），让模型学会把正样本拉近、把负样本推远。

最常用的损失函数是 **InfoNCE（也称 NT-Xent）**。对于锚点样本 $q$、正样本 $d^+$、一批负样本 $\{d^-_i\}$，目标是最大化：

$$
\mathcal{L} = -\log \frac{\exp(\text{sim}(q, d^+)/\tau)}{\exp(\text{sim}(q, d^+)/\tau) + \sum_i \exp(\text{sim}(q, d^-_i)/\tau)}
$$

其中 $\text{sim}$ 通常为余弦相似度，$\tau$ 是温度系数。这个公式直观上就是一个 $K+1$ 类的 softmax：在 1 个正样本和 $K$ 个负样本中，把概率质量集中到正样本上。

训练数据的质量与负样本挖掘策略，决定了模型的实际上限。几个关键点：

- **困难负样本（hard negatives）** 极其重要。随机负样本太「容易区分」，模型学不到细粒度的判别能力。构造「看起来相关但语义不同」的负样本（比如同一问题的不同答案、同义词的近义但不同指代的短语）能显著提升判别力。BGE 系列就大量使用了挖掘困难负样本的多阶段训练。
- **批内负样本（in-batch negatives）** 是常见技巧：一个 batch 里其他样本的正样本自动充当当前样本的负样本，从而用小批量也能构造大量负对，这是 SimCSE 等方法的基础。
- **非对称检索** 的训练范式。查询通常短、文档通常长，二者分布不同。因此许多模型（如 E5、BGE、GTE）在训练时给查询和文档分别加不同前缀（如 `query:` / `passage:`），让模型学到非对称的映射。这直接催生了下一节要讲的「指令型嵌入」。

## 三、维度：表达能力、存储与检索速度的权衡

维度 $d$ 是嵌入模型最显眼的工程参数，也是误用最多的一个。常见值从 384、768、1024 到 1536、3072 不等。维度选择是一个三向权衡：

- **表达能力**。维度越高，理论上能编码的语义信息越丰富。低维模型在细粒度区分（如同义词反义词、专业术语）上往往力不从心。
- **存储成本**。向量库中每个 chunk 存一个 $d$ 维向量。3072 维 float32 单条就占 12 KB，百万级文档便是 12 GB，且 ANN 索引（HNSW 的图结构）会进一步放大内存占用。
- **检索速度**。点积与余弦相似度的计算复杂度是 $O(d)$，维度翻倍意味着单次相似度计算翻倍；HNSW 的候选集遍历也会随之变慢。在低延迟场景（毫秒级召回）下，维度的影响会被放大。

一个常被忽略的事实是：**许多新模型支持「降维」**。OpenAI `text-embedding-3` 系列原生训练在 3072 维，但 API 支持 `dimensions` 参数，通过 Matryoshka 表示学习（MRL）在输出阶段截断到 1024、512 甚至 256 维而几乎不损失效果——因为高维信息是按重要性分层编码的，截掉末尾维度相当于丢弃最不重要的方向。SiliconFlow 的嵌入接口同样支持 `dimensions` 与 `truncate` 参数。这意味着「高维训练、低维部署」成为可行策略：用满维度训练保证上限，按场景降维部署压低延迟与存储。

工程上的经验取舍：

| 场景 | 推荐维度 | 理由 |
|---|---|---|
| 原型验证、低成本试错 | 384–768 | 存储与延迟最低，效果够用 |
| 生产通用 RAG | 1024–1536 | 表达力与成本平衡点 |
| 高精度垂直领域 | 3072 或原生高维 | 榨干语义判别力，配合量化降存储 |
| 边缘/移动端 | 256–512 | 内存与算力严格受限 |

## 四、归一化与相似度度量

有了向量，怎么算「相似」？三种度量最常见：**点积、余弦相似度、欧氏距离**。三者的关系比想象中紧密：

- **余弦相似度** $\cos(a,b) = \frac{a \cdot b}{\|a\|\|b\|}$，只关心方向，对向量模长不敏感，是语义检索的事实标准。
- **点积** $a \cdot b = \|a\|\|b\|\cos\theta$，含模长信息。当两边都归一化（$\|a\|=\|b\|=1$）后，点积等价于余弦相似度。
- **欧氏距离**关心空间中的绝对距离，对模长敏感；归一化后它与余弦相似度单调相关。

由此得出一个关键工程结论：**只要对所有向量做 L2 归一化（除以自身范数），点积、余弦、欧氏距离三者排序结果完全一致**，而点积计算最快（无需开方、无需除范数）。这就是为什么大多数向量库（包括 pgvector 的 `vector_ip`、FAISS 的 `IndexFlatIP`）都默认用归一化 + 内积来近似余弦检索。

但要注意两点：

1. **是否需要归一化取决于模型**。有些模型（如 OpenAI、BGE）默认输出已近似归一化但并非严格单位向量，工程上仍建议显式归一化以保证数值稳定；有些模型输出刻意保留模长作为「置信度/信息量」信号，此时归一化会丢信息。查阅模型文档而非想当然。
2. **混合检索时度量必须统一**。当稠密向量与 BM25 分数融合（如本项目的 RRF 融合）时，稠密侧用归一化余弦即可；但如果要分数加权融合而非 RRF，必须把两路分数缩放到同一量纲，否则一路会压过另一路。

## 五、主流嵌入模型对比

下表横向对比当前 RAG 工程中最常用的几类嵌入模型。

| 模型 | 来源 | 典型维度 | 中文表现 | 部署方式 | 特点 |
|---|---|---|---|---|---|
| text-embedding-3-small | OpenAI | 1536（可降维） | 良好 | API | 性价比高，MRL 降维灵活 |
| text-embedding-3-large | OpenAI | 3072（可降维） | 优秀 | API | 顶级效果，成本与延迟较高 |
| bge-large-zh / bge-m3 | 智源 BAAI | 1024 / 1024 | 顶尖 | 自托管/API | 中文专项优化，m3 支持稠密+稀疏+多向量 |
| bge-m3 (multilingual) | 智源 BAAI | 1024 | 优秀 | 自托管 | 多语言、长文本（8192 token），三种检索模式 |
| SiliconFlow 嵌入 | 硅基流动 | 可配 | 优秀 | API | 聚合 BGE/Qwen 等国产模型，国内访问稳定 |
| M3E | Moka AI | 768 | 优秀（早期标杆） | 自托管 | 中文社区经典基线，已较少更新 |
| jina-embeddings v3 | Jina AI | 1024（可降维） | 良好 | API/自托管 | 长文本强，支持 LoRA 任务适配 |
| gte-large-zh | 阿里达摩院 | 1024 | 优秀 | 自托管 | 中文表现稳定，适合私有化 |

几点工程判断：

- **OpenAI 系列是「开箱即用」的稳妥默认值**，尤其在英文与多语言混合场景。它的 MRL 降维能力让 3072 维模型可以按需裁到 1024，兼顾上限与成本。缺点是依赖外网与计费，且对中文长文本不如国产专项模型。
- **BGE 系列是中文场景的首选**。bge-large-zh 在中文检索榜单长期领先，bge-m3 更进一步支持 8192 token 长上下文，并原生提供稠密、稀疏、多向量（ColBERT 风格）三种检索模式，是当前最全能的开源中文嵌入之一。
- **SiliconFlow 是国内团队的折中方案**：它本身不训练模型，而是以统一 API 聚合 BGE、Qwen 等国产模型，免去自托管运维，国内访问延迟与稳定性优于直连 OpenAI，适合不想自建 GPU 服务又受限于网络合规的团队。
- **M3E 仍是中文社区的经典基线**，768 维、轻量、效果可靠，适合做对比实验的下界参照，但因迭代放缓，新项目更推荐直接用 BGE。
- **jina 在长文本场景有优势**，配合其检索模型可处理整篇文档级嵌入，适合「不分块直接检索长文档」的实验性场景。

## 六、中文场景选型

中文 RAG 的选型有几个特有的考量：

1. **分词与字符级建模**。中文没有天然空格，模型的 tokenizer 设计直接影响对短查询和专业术语的覆盖。国产模型（BGE、GTE、M3E）针对中文优化了 tokenizer，在成语、专业术语、数字混合文本上表现更稳。
2. **混合语料**。如果语料中英文混排（技术文档、产品规格），优先选 multilingual 模型如 bge-m3，而非纯中文模型——纯中文模型对英文专有名词（API 名、产品型号）的区分度往往不足。
3. **长文本**。bge-m3 支持 8192 token，能让一个 chunk 更大、减少信息割裂；但要注意长 chunk 会稀释相似度信号，需配合合理的分块策略。
4. **私有化与合规**。金融、政务、医疗等场景要求数据不出域，必须自托管，OpenAI/SiliconFlow API 不可用，应选 BGE、GTE 等可本地部署的开源模型。

一个简明的中文选型决策树：

- 需要私有化部署 → **bge-m3**（全能）或 **bge-large-zh**（轻量）
- 可用 API、追求省心 → **SiliconFlow**（国内）或 **OpenAI text-embedding-3-large**（海外）
- 长文本为主 → **bge-m3** 或 **jina v3**
- 英文/多语言为主 → **OpenAI text-embedding-3** 或 **bge-m3**

## 七、指令型嵌入（Instruction Embedding）

指令型嵌入是近两年的重要进展，源自 E5 等模型开创的「前缀任务化」思路，并在 InstructOR、BGE、GTE 等模型中发扬光大。其核心是：**在文本前拼接一段任务指令前缀**，让同一个模型服务于多种检索任务，而不必为每种任务单独微调。

典型用法：

- 查询侧加 `query: ` 或 `Represent this sentence for searching relevant passages: `
- 文档侧加 `passage: ` 或不加前缀

例如 BGE 系列官方推荐查询侧加 `为这个句子生成表示以用于检索相关文章：` 这样的中文指令。指令的加入让模型「知道」当前是在做检索匹配，而非普通的句子编码，从而调整嵌入分布以更贴合检索目标。

工程要点：

- **指令必须与训练时一致**。模型在训练时用什么指令，推理时就要用同样的，否则效果会下降甚至劣化。换指令等于换了分布。
- **查询与文档的非对称前缀**。这是非对称检索（短查询检索长文档）能work的关键，文档侧通常不加或加 `passage:`。
- **并非所有模型都支持**。OpenAI text-embedding-3 不需要也不支持指令前缀，强行拼接反而可能引入噪声；BGE/E5/GTE 则强烈建议按官方文档加。务必查阅对应模型的用法说明。

## 八、稠密 vs 稀疏嵌入

稠密嵌入（dense embedding）是上述 $d$ 维连续向量，擅长语义匹配（同义不同字），但对罕见专有名词、产品型号、代码标识符这类「字面敏感」的查询容易召回失败——因为它们被压进了一个语义平滑空间，稀有词的细节被稀释。

稀疏嵌入（sparse embedding）则相反，它更像一个可学习的 BM25：向量维度极高（词表规模），大部分维度为 0，少数维度有值，保留了字面匹配能力。SPLADE 是典型代表。

两者各有短板，现代 RAG 趋势是**混合检索（hybrid retrieval）**：

- 稠密负责语义召回（「怎么提高系统性能」匹配到「优化响应速度」）；
- 稀疏/BM25 负责字面召回（精确匹配型号、错误码、API 名）；
- 两路结果用 RRF（倒数排名融合）或加权分数融合。

bge-m3 一个模型的重大价值就在于它原生同时输出稠密、稀疏、多向量三种表示，免去维护三个模型。本项目 `VectorIndexService` 也实现了 vector + bm25 的 hybrid 模式并用 RRF 融合，正是这一思路的落地。

## 九、嵌入更新与版本兼容问题

嵌入模型不是一锤子买卖，长期运行必然面临版本演进，这也是最容易踩坑的运维问题：

1. **维度与语义空间不可跨版本兼容**。即使是同一模型系列的不同版本（如 `text-embedding-ada-002` → `text-embedding-3-large`，或 `bge-large-zh-v1` → `v1.5`），其输出向量的语义空间完全不同，维度也可能变化。新旧向量混在一个库里检索毫无意义——相似度会给出误导性结果。
2. **迁移策略**。换模型必须**全量重建索引**：要么停服重灌，要么双写双查灰度切换（新库与旧库并存，逐步切流量，全量重建完成后下线旧库）。绝不能「新文档用新模型、旧文档留旧模型」地增量混用。
3. **持久化元数据**。每个向量记录必须带上 `provider` 与 `model`（以及 `dimensions`），以便后续判断是否需要重建。本项目 `VectorIndexService.status()` 正是返回 `embeddingClient.providerName()` 来暴露当前索引所用 provider，`VectorRecord` 也携带 `indexedAt` 时间戳——这些都是版本治理的基础。
4. **降级兼容**。当外部嵌入服务不可用时，系统应有降级方案。本项目的 `HashingEmbeddingClient` 即是这种角色：它用哈希特征 + L2 归一化生成确定性向量，无需任何外部依赖，可在无 API key、离线或服务故障时让系统继续运转（虽然语义质量远不如真模型，但保证了管线不中断）。

## 十、本项目对照：EmbeddingClient 多 provider 抽象

本项目的 storage-layer 通过一个简洁的接口抽象了嵌入客户端，使上层检索逻辑（`VectorIndexService`）与具体 provider 解耦：

```java
public interface EmbeddingClient {
    List<float[]> embed(List<String> texts);
    default float[] embedOne(String text) { ... }
    String providerName();
}
```

`embed` 批量编码文本返回浮点向量，`providerName` 暴露 provider 标识（用于状态展示与版本治理）。该接口有三个实现，通过 Spring 的 `@ConditionalOnProperty` 按 `rag.vector.embedding.provider` 配置值条件装配，运行时只会激活一个：

- **`SiliconFlowEmbeddingClient`**（默认，`matchIfMissing = true`）：调用硅基流动 `/v1/embeddings`，请求体携带 `model`、`dimensions`、`encoding_format`、`truncate` 等参数，支持降维与截断，按 `index` 排序保证返回顺序与输入一致。
- **`OpenAiEmbeddingClient`**（`provider = openai`）：调用 OpenAI `/v1/embeddings`，结构更简（仅 model + input），适合海外或合规允许的场景。
- **`HashingEmbeddingClient`**（`provider = hash`）：纯本地哈希特征嵌入，无任何外部依赖，作为降级与离线方案。

三者均实现 L2 归一化（Hashing 显式归一化，OpenAI/SiliconFlow 的归一化策略取决于模型输出），落盘到 `MemoryVectorStore`（开发/测试）或 `PgVectorStore`（生产，基于 pgvector）。`VectorIndexService` 在 `indexDocument` 时批量编码 chunk 写入向量库，在 `search` 时批量编码（经查询扩展后的多条）查询向量分别检索，再与 BM25 结果 RRF 融合。这一抽象让「换模型」变成改一行配置 + 全量重建索引的纯工程动作，不触碰任何业务代码。

## 小结与选型建议

文本嵌入是把语义映射进高维空间的压缩函数，其质量决定 RAG 检索上限。要点回顾：

- **本质**：文本到 $\mathbb{R}^d$ 的语义映射，查询与文档必须同模型、同归一化。
- **训练**：对比学习 + InfoNCE，困难负样本是效果分水岭，非对称前缀催生指令型嵌入。
- **维度**：高维表达强但贵，可用 MRL 降维在「高维训练、低维部署」间平衡。
- **度量**：归一化后点积等价余弦且最快，混合检索需统一量纲或用 RRF 融合。
