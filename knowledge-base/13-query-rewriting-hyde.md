# 查询重写与语义增强：扩展、分解与 HyDE

## 一、为什么原始 query 不够

在工业级 RAG 系统里，用户输入的原始 query 直接进入检索环节，往往命中率惨淡。这并非检索器或 Embedding 模型不够强，而是因为自然语言查询天然带有四类"噪声"：

1. **口语化表达**：用户说"东西坏了咋整"，而知识库里写的是"产品故障后的维修与报修流程"。词面几乎无重叠，稠密检索勉强能靠语义近似捞回一部分，稀疏检索（BM25）几乎为零。
2. **模糊与缺关键词**：如"还有吗""能退吗""多少钱"，query 过短，关键词稀疏，Embedding 向量在语义空间里落到一个高熵区域，top-k 召回与问题相关度低。
3. **指代与省略**：多轮对话里用户说"它的保修多久"，"它"指代上一轮提到的某个产品。脱离对话历史的 query 完全无法定位实体。
4. **多意图叠加**：如"这个产品多少钱，保修多久，支持退货吗"，一句话含三个子问题。单一 query 检索要么只能命中其中一个，要么返回一个"四不像"的混合文档。

查询重写（Query Rewriting）的目标，就是在原始 query 与检索器之间插入一个语义增强层，把"用户怎么说"转译成"检索器最容易命中"的形式。它属于查询处理（Query Processing）阶段，位于意图识别之后、向量/关键词检索之前，是决定召回质量的上游阀门。

## 二、查询扩展（Query Expansion）

查询扩展的核心思想是：一个 query 不只拿去检索一次，而是派生出多个语义相关的子查询，多路召回后取并集或加权融合。

### 2.1 同义词与相关词扩展

通过词典（如领域同义词表）或 LLM 生成同义词、近义词、相关实体，拼成多个查询。例如原始 query "耳机降噪"，扩展为"主动降噪 ANC""耳机降噪原理""TWS 降噪耳机对比"等。每条扩展查询独立检索，结果合并去重。

### 2.2 LLM 多查询生成（Multi-Query）

让 LLM 针对同一意图生成 N 个不同视角的 query，覆盖不同表述方式。优点是覆盖面广，能捞回稠密检索因为表述差异而漏掉的文档；缺点是 N 路检索的延迟与成本线性增长，且容易引入噪声（扩展查询偏离原意导致误召回）。

### 2.3 适用场景

- query 过短、关键词稀疏时，扩展能显著提升召回。
- 领域术语存在多表述（如医学、法律领域），同义词扩展尤其有效。
- 需配合去重与重排，否则噪声文档会拉低最终答案质量。

## 三、查询分解（Query Decomposition）

当 query 包含多个子问题时，与其用一个混合 query 检索，不如拆成若干子问题，各自检索，再合并证据喂给生成器。

### 3.1 分解策略

- **规则分解**：按分隔符（，,；;另外 顺便 以及 和）切分，再做语义校验。实现简单、延迟低，但对自然连贯的长句无效。
- **LLM 分解**：让 LLM 把复杂问题拆成原子子问题，并判断每个子问题是否需要独立检索。更灵活，但增加一次 LLM 调用。

### 3.2 合并策略

分解后每路检索的文档，可以简单拼接去重，也可以按子问题做加权（与原始意图相关性高的子问题结果权重更高），再统一进重排（rerank）。需注意：子问题之间可能存在依赖（后一个子问题依赖前一个的答案），此时需要顺序检索而非并行。

### 3.3 与扩展的区别

扩展是"同一问题、多个表述"，目的是扩大召回面；分解是"一个问题、多个子问题"，目的是把复杂检索拆成可独立命中的原子单元。两者可以叠加：先分解，再对每个子问题做扩展。

## 四、HyDE：假设性文档嵌入

HyDE（Hypothetical Document Embeddings）是一种巧妙的语义增强思路：与其用用户的简短问题去检索，不如让 LLM 先"假装回答"这个问题，生成一段假设性答案（假设性文档），再用这段答案的 Embedding 去检索真实文档。

### 4.1 原理

直觉来自一个观察：**文档与文档之间的语义相似度，往往高于问题与文档之间的相似度**。用户问题是疑问句、短、口语化；知识库文档是陈述句、长、书面化。两者在 Embedding 空间本就有分布差异（query-document gap）。HyDE 通过生成一段"长得像答案"的假设文档，把检索从"问题-文档"匹配拉近到"文档-文档"匹配，从而缩小这一分布差距。

流程为：原始 query → LLM 生成假设性答案（不要求事实正确，只要语言风格、领域术语接近真实文档）→ 对假设答案做 Embedding → 用该向量检索知识库 → 检索到的真实文档作为最终证据喂给生成器。

### 4.2 优劣

- **优点**：无需额外训练，直接利用 LLM 的生成能力；对短 query、术语缺乏的 query 提升明显；能捕获 LLM 先验中蕴含的领域知识。
- **缺点**：多一次 LLM 生成，延迟与成本显著增加；假设答案若方向偏差（LLM 幻觉），可能把检索引向错误区域；生成内容需做长度与质量校验。

### 4.3 适用与不适用

- 适用：事实型问答、query 与文档表述差异大的场景。
- 不适用：用户意图本身模糊（LLM 难以生成合理假设）、对延迟敏感的实时对话、需要精确关键词匹配（如编号、订单号查询，此时 HyDE 反而引入噪声）。

## 五、查询纠错与规范化

在重写之前，先做一轮轻量的文本规范化，能消除大量低级噪声：

- **空白与标点规范化**：合并多余空格、全半角统一、去除首尾噪声。
- **拼写与输入纠错**：针对用户手误（如"耳几"→"耳机"），可用编辑距离或轻量模型纠错。
- **口语化形式化**：把"东西坏了咋整"映射为"产品故障后的维修和报修流程"，可用规则映射或 LLM 改写。
- **实体归一**：产品型号、专有名词的别名归一。

规范化是成本最低、收益最稳的一步，应在所有重写策略之前执行。

## 六、重写策略对比

| 策略 | 触发条件 | 核心手段 | 召回影响 | 额外延迟 | 成本 |
|------|----------|----------|----------|----------|------|
| 规范化 | 所有 query | 文本清洗、口语形式化 | 平稳提升 | 极低 | 极低 |
| 指代消解 | 含代词/短追问+历史 | 结合历史补全实体 | 显著提升 | 低（可规则） | 低 |
| 查询扩展 | query 过短/关键词稀疏 | 同义词、多查询生成 | 扩大召回面 | 中（多路检索） | 中 |
| 查询分解 | 多意图叠加 | 拆子问题分路检索 | 提升多意图命中 | 中高 | 中 |
| HyDE | 事实型、query-document 差距大 | 生成假设文档再嵌入 | 缩小分布差距 | 高（多一次生成） | 高 |

选型原则：**按需触发，而非全量套用**。简单明确的 query 应原样透传，避免画蛇添足。

## 七、重写后查询与原始查询的融合检索

重写并非简单地"替换"原始 query，更稳妥的做法是**融合**：重写后的 query 用于召回，原始 query 也保留一路参与检索，结果合并。原因有二：

1. 重写可能引入偏差（LLM 改写偏离原意、假设文档方向错误），保留原始 query 是一道安全网。
2. 原始 query 中的精确关键词（产品编号、专有名词）可能在改写中被弱化，保留一路能确保精确匹配不丢。

融合方式可以是多路召回后 RRF（Reciprocal Rank Fusion）合并，或对各路结果加权再 rerank。重写后查询与原始查询的权重配比需根据线上效果调参，常见做法是重写查询权重略高于原始查询，但不让原始查询权重归零。

## 八、延迟与成本考量

查询重写本质是用上游的计算换下游的召回质量，必须量化其代价：

- **延迟**：每多一次 LLM 调用，首字延迟增加数百毫秒到数秒。HyDE 与 LLM 多查询生成是延迟大头，对实时对话需谨慎或采用流式/异步。
- **成本**：LLM 调用按 token 计费，重写 prompt 加历史上下文，单次成本不容忽视。高 QPS 场景下，重写是显著的费用来源。
- **缓存**：相同（query + 历史）的重写结果可缓存复用，对热点问题尤其有效。
- **降级**：LLM 重写失败时应降级到规则重写或直接透传，保证链路可用性，不能因重写服务异常拖垮整个对话。

实践建议：对低价值 query（短、清晰、无指代）跳过重写；对高价值 query（含指代、多意图、口语化）才走完整重写链路，把 LLM 调用花在刀刃上。

## 九、本项目对照：query-rewrite-service

在本项目 RAG 链路中，查询重写由独立的 **query-rewrite-service** 承担，位于 agent-service 之下、检索之前。agent-service 在收到用户输入后，调用 query-rewrite-service 完成前端 query 的分析、意图分类、路由建议与 query rewrite，再依据返回结果决定后续检索与生成路径。

该服务对外暴露两个接口（端口 28082）：

- `POST /api/chat/analyze`：返回完整的查询分析，包括原始 query、规范化 query、改写 query、意图、置信度、路由建议、是否需要改写、历史长度、检索用 queries 与原因列表。
- `POST /api/chat/query-rewrite`：返回精简的改写结果，含改写后 query、意图、置信度、路由与 retrievalQueries。

实现上体现了上文的多项策略：

- **规范化**：`normalize` 做空白合并与首尾清理，是所有处理的起点。
- **意图路由**：非知识检索意图（如闲聊）跳过重写，原样透传，避免无效改写。
- **按需改写**：`shouldRewrite` 依据是否含指代词（它、这个、那个等）、是否短追问带历史、是否口语化、是否多意图等信号判定，只有命中才触发 LLM 改写。
- **LLM 改写 + 规则降级**：优先用 LLM 结合对话历史补全指代、形式化口语；LLM 不可用或返回异常时降级到本地规则改写（`rewriteWithLocalContext`，基于历史抽取实体做指代替换与短追问补全）。
- **多意图分解**：`decomposeQueries` 对多意图 query 按分隔符切分为子问题，形成多路 retrievalQueries 供下游检索。
- **结果可解释**：每一步决策都写入 reasons 列表（如 contains_coreference_marker、rewrite_applied、multi_intent_decomposed），便于排查与调优。

值得注意的工程取舍：本项目尚未实现 HyDE，因为其额外 LLM 生成成本与延迟在当前对话场景下不划算；而是选择了更轻的"指代消解 + 口语形式化 + 多意图分解"组合，配合按需触发与规则降级，在召回质量与链路稳定性之间取得平衡。若后续面对短 query、事实型问答占比升高的场景，可考虑引入 HyDE 作为可选增强路径，并通过意图路由仅在合适场景下启用。

## 十、小结

查询重写是 RAG 检索质量的"前置放大器"。原始 query 因口语化、模糊、指代、多意图而难以直接命中，重写层通过规范化、扩展、分解、HyDE 等手段把 query 转译为检索器友好的形式。核心原则是：**按需触发、规则与 LLM 分级降级、重写与原始查询融合而非替换、严控延迟与成本**。HyDE 以"生成假设文档再嵌入"缩小 query-document 分布差距，效果显著但代价偏高，应作为可选增强而非默认开启。本项目的 query-rewrite-service 即为这一思想的落地：在 agent-service 调用链中承担意图分类、路由建议与按需改写，通过指代消解、口语形式化与多意图分解提升召回，同时以规则降级与 reasons 可解释性保障链路稳定。
