# 引用（Citation）与可溯源：来源标注与可信回答构建

## 一、为什么引用对 RAG 可信度是关键

RAG 系统的核心承诺是「答案来自你提供的知识库，而不是模型凭空捏造」。但这个承诺在用户侧其实无法被直接验证——用户看到的是一段自然语言文字，并不知道其中哪一句来自哪个文档、哪一句是 LLM 在「转述」过程中加进去的演绎。如果一段看起来专业、笃定的回答其实并没有对应的知识库依据，用户却照单全收去决策，风险就被悄悄放大了。

**引用（Citation）就是把「黑盒回答」变成「可审计回答」的机制**。它要求回答中的关键论断必须挂载到具体的来源片段上，用户可以一眼看出「这句话出自哪份文档的哪一段」，甚至点进去核对原文。这个机制带来的不是单纯的「体验加分」，而是三个层面的可信度提升：

- **可验证性**：用户能在原文里确认模型没有偷换概念、没有断章取义，回答是知识库的真实反映。
- **可归责性**：当答案出错时，可以快速定位是「检索没召回到正确片段」「片段本身就写错了」还是「LLM 没忠实转述」，而不是把锅全甩给「模型不行」。
- **可治理性**：幻觉（hallucination）的治理有了一个可度量的抓手——无引用的回答、引用内容与原文不符的回答，都可以被检测、被拦截、被纳入评估指标。

一句话：**没有引用的 RAG，本质上是把一个检索增强的生成器伪装成了一个可信的知识问答系统**。引用是可信回答的地基。

## 二、引用的实现方式：从标注粒度到来源标识

引用的实现涉及两个独立的设计维度：一是「在回答里怎么标注」，二是「标注的指针指向什么」。

### 2.1 标注粒度方案对比

| 方案 | 标注形式 | 优点 | 缺点 | 适用场景 |
|---|---|---|---|---|
| 行内标注 | `[1]`、`[1,2]` 紧跟句子末尾 | 粒度细，逐句可溯源；学术写作习惯，用户易读 | LLM 不总是稳定产出编号；编号易错位 | 事实型问答、政策条款问答 |
| 脚注式标注 | 句末上标 `¹²`，文末列引用表 | 与正文分离，阅读流畅 | 上标在纯文本/语音通道里易丢失 | 长文综述、报告生成 |
| 段落级标注 | 每段末尾 `本段来源：[3][4]` | 实现简单，LLM 易遵循 | 粒度粗，段内多论断无法区分来源 | 摘要型回答、轻量场景 |
| 无标注+来源列表 | 仅文末给出来源清单 | 实现成本最低 | 无法判断哪句对应哪个来源，可溯源性最弱 | 不推荐，仅作兜底 |

工程上最常用、性价比最高的是**行内标注 `[1]`**：它既保留了细粒度，又和「检索上下文按编号拼接」的实现天然契合——只要在拼 prompt 时给每个 chunk 打上 `[1]`、`[2]` 的编号，模型就倾向于复用这些编号。

### 2.2 来源标识：引用指针指向什么

标注的编号最终要解析回一个真实来源。指针的「信息密度」决定了可溯源链路能走多远：

- **仅文档名**：`[1] 产品需求文档.pdf`。能定位到文档，但文档可能几十页、上千 chunk，无法定位到具体位置。适合文档都很短、或只需要「出处是哪份文件」的场景。
- **文档名 + 页码**：`[1] 产品需求文档.pdf, p.4`。对 PDF 类文档友好，前端可实现「点击跳转到第 4 页高亮」。但对按语义切片的 markdown/txt 无意义，因为 chunk 并不对应固定页。
- **chunk id**：`[1] chunk=doc-3-#7`。直接指向知识库里的某个切片，可反查原文内容、所在文档、在文档内的偏移。这是 RAG 系统最本质、最稳定的指针，因为切片本身就是检索和生成的基本单元。
- **复合标识**：`documentId + chunkId + chunkIndex + documentName`。兼顾「对人可读」和「对程序可定位」。

实践中推荐**以 chunkId 为主键、documentName + chunkIndex 作为展示**：程序内部用 chunkId 反查原文（稳定、唯一），展示给用户时渲染成「产品需求文档.pdf / 第 7 段」这种可读串，点击再跳转到原文。

## 三、如何让 LLM 稳定产出引用

引用机制最大的工程难点不在「拼接」，而在「让模型听话」——LLM 天然倾向于生成流畅但无标注的文字，甚至把多个片段揉成一句后丢掉编号。要稳定产出引用，需要**Prompt 约束 + 上下文编号化**双管齐下。

### 3.1 上下文编号化

在拼装检索上下文时，给每个 chunk 显式编号，并附带可定位信息。典型格式如下：

```
知识库片段：
[1] 文档：产品需求文档.pdf，chunk：doc-3-#7，分数：0.8312
<chunk 原文内容>

[2] 文档：入职流程.docx，chunk：doc-5-#2，分数：0.7956
<chunk 原文内容>
```

编号一旦写进上下文，模型就有了可复用的「引用词典」。它不需要自己去发明编号，只需在引用时复用 `[1]`、`[2]`。这一步看似简单，却是稳定引用的物理基础——**没有编号化的上下文，再强的 Prompt 约束也压不住模型自由发挥**。

### 3.2 Prompt 约束要点

系统提示里应明确以下几条规则：

1. **优先依据提供的片段回答**，不能从片段推出的内容要明确说明「没有依据」。
2. **引用片段内容时使用 `[1]`、`[2]` 编号**，编号来自上下文。
3. **答案依赖某片段时，在对应句子后标注引用编号**（而非全文末尾统一标注）。
4. **不要编造文档名、条款号或编号**——即不得出现上下文里没有的 `[3]`。
5. 不确定时宁可不答，也不要用看似权威的语气补全。

其中第 4 条尤其关键：模型有时会「自信地」引用一个上下文中并不存在的编号（幻觉式引用），需要在 Prompt 层面预先约束，并在后处理阶段做校验。

### 3.3 稳定性技巧

- **few-shot 示例**：在 Prompt 里给 1-2 个「问题→带引用回答」的示例，示范编号如何挂到具体句子后，比纯规则更有效。
- **限制 top-K 数量**：引用编号过多（如 `[1]` 到 `[12]`）时模型更容易漏标或错位。把进入上下文的 chunk 数控制在 4-6 个，编号空间小、模型遵循率显著上升。
- **答案与引用分离生成**：先让模型生成正文，再单独让它「为每个引用编号回填一句话原文摘要」做交叉校验。开销略大但准确率最高，适合高可信场景。

## 四、引用准确性校验

模型「听话地」产出了 `[1]`，并不等于这个 `[1]` 真的支撑了那句话。引用准确性校验就是在生成之后做一道质检，回答两个问题：**引用编号是否合法？引用内容是否真的在 cited chunk 中？**

### 4.1 编号合法性校验

最基础的校验：扫描回答中所有 `[n]` 标记，确认每个 `n` 都落在上下文提供的编号集合 `{1,2,…,K}` 内。超出范围的编号（如上下文只有 5 个 chunk 却出现了 `[7]`）属于幻觉式引用，应直接标记为低可信或触发重生成。

### 4.2 引用-原文一致性校验

更进阶的校验是判断「标注了 `[1]` 的这句话，是否真的能从 chunk 1 的原文中得出」。这又分两层：

- **字面包含**：把 `[1]` 所在句子的关键短语与 chunk 1 原文做包含/重叠匹配。粗粒度但实现成本低，能抓住「张冠李戴」（把 A 片段的内容标成了 B 的编号）这类明显错误。
- **语义蕴含**：用 NLI（自然语言推理）模型或另一个 LLM 作为 judge，判断「chunk 原文 ⇒ 该论断」是否成立。粒度更细，能识别「原文说的是上限，回答却讲成了下限」这类语义性幻觉。建议对关键论断（数字、日期、政策条款、医疗/法律结论）强制做语义校验。

### 4.3 引用覆盖率与引用精度

把引用准确性纳入评估指标，可定义两个互补的指标：

- **引用覆盖率（Citation Recall）**：答案中「应当有引用的事实性论断」里，实际带引用编号的比例。低覆盖率说明模型大量「裸答」，可信度存疑。
- **引用精度（Citation Precision）**：所有标注的引用里，经校验确实能从对应 chunk 推出的比例。低精度说明模型在「乱挂引用」。

这两个指标应纳入 RAG 评估体系的回归集，作为上线和迭代的质量门槛——无引用或引用不准的答案，宁可降级为「未找到相关依据」也不直接吐给用户。

## 五、可溯源链路设计：chunk → 文档 → 原文位置

引用的本质是「指针」，而指针的价值取决于它能否被一步步解析到原文。一个完整的可溯源链路应该是：**回答中的 `[n]` → 检索命中的 chunk（chunkId）→ 所属文档（documentId/documentName）→ 文档内的原文位置（页码/偏移/章节）→ 原文片段高亮展示**。

要让这条链路走得通，有几个关键工程约定：

- **chunk 持久化原文内容与定位元数据**：每个 chunk 在入库时除了存向量和正文，还应记录 `documentId`、`chunkIndex`、在原文中的字符偏移（或页码/坐标）。这是可溯源的数据基础，缺失则链路断在第一步。
- **编号 ↔ chunkId 的映射在生成时即确定**：拼上下文时 `[n]` 对应哪个 chunkId 是固定的，生成完成后这个映射应随答案一起返回，而不是事后再去检索还原。
- **后端返回结构化 citations 列表**：前端不应靠解析回答正文里的 `[n]` 来反推来源，而应由后端在响应里直接给出结构化的引用列表（每条含编号、文档名、chunkId、分数、摘要），前端只负责渲染和跳转。
- **摘要字段用于预览**：citation 里带一段 200 字左右的原文 excerpt，前端可在不请求原文的情况下做悬停预览，降低「点开看全文」的成本。

这条链路把「回答」从一段孤立文字，变成了一个锚定在知识库实体上的可追溯对象。任何一句论断，都能顺着链路回到它的原文出处。

## 六、引用与幻觉治理的关系

幻觉治理是 RAG 的核心命题，而引用是其中「可检测、可拦截」的那一环。两者关系可以从三个层面理解：

- **引用是幻觉的探测器**。无引用的事实性论断是幻觉的高发区——模型「无中生有」的内容必然没有合法编号。统计回答里「事实句带引用率」就是一个粗粒度幻觉率指标。
- **引用是幻觉的拦截器**。在校验环节，编号非法、引用-原文不一致都可直接拦截或触发重生成，把幻觉挡在用户看到之前。这比事后用 LLM-judge 笼统打分更精准、更便宜。
- **引用是幻觉治理的反馈源**。长期统计「哪类问题、哪个知识库、哪个 chunk 容易被错引」，能反过来定位检索质量短板（如召回率低导致模型只能靠编造）或切片质量问题（如 chunk 切断了关键语境，导致模型误读）。

需要强调：引用不是幻觉的银弹。模型完全可能产出一个「合法编号、却与原文语义不符」的回答——引用存在且编号合法，但论断其实是错的。因此引用治理必须配合语义一致性校验，二者叠加才能把幻觉压到可接受水平。

## 七、前端引用展示与交互

引用的可信价值最终要在前端兑现。一个好的引用交互应满足「可见、可预览、可跳转、可对照」：

- **行内编号渲染**：把正文里的 `[1]` 渲染成可点击的小徽章（带上标或色块），与正文视觉区隔，鼠标悬停显示来源摘要。
- **悬停预览 excerpt**：鼠标移到编号上，弹出该 chunk 的文档名、分数和一段原文摘要，让用户不必跳转就能初判是否靠谱。
- **点击跳转原文**：点击编号滚动到答案下方/侧栏的「引用来源」面板对应条目；条目可进一步点击打开原文（PDF 跳到对应页、markdown 跳到对应 chunk），并高亮命中片段。
- **来源面板与答案联动**：右侧或底部常驻「引用来源」列表，列出本次回答的所有 citations（文档名、页码/chunk 号、相似度分数），鼠标悬停某条时答案中对应编号高亮，反之亦然。
- **无引用提示**：若答案事实句未带任何引用，前端应给出视觉提示（如「本回答未标注来源」灰色标记），把可信度判断权交还用户。

## 八、本项目对照

本项目的引用链路体现了上述设计的核心思路，由 `agent-service` 生成结构化引用、`frontend` 负责展示。

**检索与编号阶段**：`ChatOrchestrator.retrieve()` 拿到向量检索命中的 `VectorSearchMatch` 后，按分数降序、去重、截断 top-K，为每条命中分配自增 `index`（从 1 开始）并封装为 `RetrievalHit`。`RetrievalHit` 携带 `knowledgeBaseId`、`documentId`、`chunkId`、`chunkIndex`、`documentName`、`content`、`score` 全套定位字段——即上文的「复合标识」。

**Prompt 编号化阶段**：`PromptBuilder.appendContext()` 把每个 hit 拼成 `[1] 文档：xxx，chunk：xxx，分数：xxxx\n<原文>` 的编号块，并受 `maxContextCharacters` 约束防止超长。系统提示明确要求「引用知识库内容时使用 [1]、[2] 这样的编号」「在对应句子后标注引用编号」，并禁止编造文档名/条款。

**结构化引用返回阶段**：`RetrievalHit.toCitation()` 把检索命中转换为 `Citation` record，关键字段保留 `chunkId`、`documentId`、`chunkIndex`、`documentName`、`score`，并额外生成一段 220 字的 `excerpt`（原文摘要，去空白后截断加省略号）供前端预览。`ChatResponse` 在返回 `answer` 文本的同时，并行返回 `citations` 列表——前端不靠解析正文编号反推来源，而是直接消费这个结构化列表。SSE 流式接口（`ChatController`）也会在 delta 推完后单独发一个 `citations` 事件，保证流式场景下引用仍能随答案送达。

**前端展示阶段**：`frontend/src/App.tsx` 定义 `ChatCitation` 类型（`index/documentName/chunkIndex/score/excerpt`），拿到响应后把 citations 映射成「文档名 / chunk N / 分数」的来源字符串列表渲染在答案下方。当前实现以来源徽章列表为主，预留了「展开引用」「来源预览（含页码·命中分数）」等交互入口。需要说明的是，本项目目前**尚未实现行内 `[n]` 点击跳转原文与引用-原文一致性自动校验**——编号合法性、引用精度等校验逻辑属于可继续补强的方向。

## 小结

引用与可溯源不是一个「锦上添花」的前端功能，而是 RAG 可信度的结构性支柱。它的工程闭环可以浓缩为五步：**检索命中编号化 → Prompt 约束模型复用编号 → 后处理校验编号合法性与引用-原文一致性 → 结构化 citations 随答案返回 → 前端渲染可预览、可跳转的来源交互**。每一步都把「黑盒回答」向「可审计回答」推进一步，最终让用户能验证、让故障可定位、让幻觉可度量。对本项目而言，编号化上下文与结构化 citation 链路已经搭好，下一步应补齐引用准确性校验与前端跳转原文，把可溯源从「展示来源」升级为「核对来源」。
