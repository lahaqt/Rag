# 重排序（Reranking）：Cross-encoder、Cohere Rerank 与二次精排的必要性

## 一、为什么需要重排序

在一个典型的 RAG 系统中，检索链路通常被拆成两个目标相互矛盾的阶段：

- **召回阶段（Recall）**：从百万乃至亿级文档库中快速筛出「可能相关」的候选集合。这一阶段的头号指标是**速度**和**召回率**——宁可多捞一些，也不能漏掉真正该命中的答案。
- **精排阶段（Precision）**：在少量候选中挑出「最相关」的若干条，喂给下游 LLM 做上下文生成。这一阶段的头号指标是**排序精度**——位置越靠前的文档，越要真正回答用户问题。

召回阶段为了速度，几乎无一例外地采用 **Bi-encoder（双编码器）**：query 和 doc 各自独立编码成一个向量，再用内积或余弦相似度计算。这种设计的好处是 doc 向量可以离线预计算并建成 ANN 索引（HNSW、IVF-PQ 等），在线查询时只需要一次 query 编码 + 一次 ANN 搜索，延迟在毫秒量级。

但 Bi-encoder 的代价是：query 和 doc 在编码时**完全不知道彼此的存在**，它们之间没有交互。这意味着：

- 词汇级别的细粒度匹配（比如 query 里的某个专业术语在 doc 里恰好出现）难以被捕捉。
- 语义上的因果、条件、否定等复杂关系容易被稀释成一个粗略的相似度分数。
- 一些「形似而神不似」的文档（同领域但答非所问）会获得虚高的分数。

结果就是：召回阶段的 top-K 里常常混入「看起来相关、但实际不回答问题」的文档，挤掉了真正该排在前面的答案。LLM 拿到这种污染过的上下文，要么答错，要么产生幻觉。

**重排序（Reranking）正是为了弥补这一缺陷**：在召回阶段拿到一个较大的候选集（top-N，N 较大）之后，用一个精度更高、但更慢的模型对这 N 条候选逐一重新打分，再按新分数排序，截取 top-K（K 远小于 N）作为最终上下文。这就是所谓的**二次精排**。

## 二、Cross-encoder 的原理

Cross-encoder 的核心思想是：**把 query 和 doc 拼接在一起，作为一个整体送入 Transformer 进行联合编码**，让两者在每一层都通过 self-attention 充分交互，最后输出一个相关性分数。

一个典型的 Cross-encoder 打分流程如下：

```
输入: [CLS] query [SEP] document [SEP]
      ↓
   Transformer Encoder (多层 self-attention)
      ↓
   取 [CLS] 的隐藏向量
      ↓
   线性层 + sigmoid
      ↓
   相关性分数 (0~1)
```

与 Bi-encoder「先编码、再算距离」不同，Cross-encoder 的相似度不是在向量空间里算出来的，而是**由模型在联合表示上直接回归出来的**。query 的每个 token 都能 attend 到 doc 的每个 token，doc 的每个 token 也能 attend 到 query 的每个 token，因此能够建模：

- 词级别的精确匹配信号（lexical match）
- query 与 doc 之间的语义对齐与蕴含关系
- 否定、条件、因果等需要跨句理解的细微语义

这带来的是显著更高的排序精度。在 BEIR、MS MARCO 等公开基准上，Cross-encoder 的 nDCG@10 通常比同规模的 Bi-encoder 高出 5~15 个点。

## 三、Bi-encoder vs Cross-encoder 对比

| 维度 | Bi-encoder（双编码器） | Cross-encoder（交叉编码器） |
|---|---|---|
| 编码方式 | query、doc 各自独立编码 | query+doc 拼接联合编码 |
| 交互层级 | 仅在最后做点积/余弦（late interaction） | 每一层 self-attention 全交互 |
| 精度 | 中等，粗粒度相似度 | 高，细粒度语义匹配 |
| 速度 | 极快，doc 向量可预计算、可建 ANN 索引 | 慢，每对 (query, doc) 都要跑一次前向 |
| 成本 | 在线仅一次 query 编码 + ANN 检索 | 在线对每个候选都要一次完整前向 |
| 可扩展性 | 可扩展到亿级文档库 | 难以对全库打分，只适合小候选集 |
| 典型用途 | 召回阶段，从全库筛 top-N | 重排阶段，对 top-N 精排到 top-K |
| 代表模型 | BGE、E5、GTE、OpenAI text-embedding | bge-reranker、jina-reranker、Cohere Rerank |

一句话总结两者的分工：**Bi-encoder 负责「又快又广地捞」，Cross-encoder 负责「又准又细地排」**。

## 四、延迟与成本代价

精度从来不是免费的。Cross-encoder 的高精度来自对每一对 (query, doc) 的完整前向计算，这带来显著的延迟和成本代价：

- **延迟**：对 N 个候选打分，需要 N 次完整的 Transformer 前向（通常 N 在几十到一两百之间）。一个中等规模的 reranker（如 110M~560M 参数）在 GPU 上单对约几毫秒，N=100 时整体重排延迟通常在 100~500ms；在 CPU 上则可能达到秒级。
- **成本**：由于无法像 Bi-encoder 那样预计算 doc 向量，每次查询都要为所有候选做计算。在调用 API 形式的 reranker（如 Cohere Rerank）时，按 token 计费，长文档会快速累积成本。
- **吞吐**：高 QPS 场景下 reranker 往往成为瓶颈，需要做批处理、并发或缓存优化。

因此，工程上必须把 Cross-encoder 限制在**一个足够小的候选集**上——这正是两阶段检索的设计依据。

## 五、何时该用重排

并不是所有 RAG 系统都必须加重排。是否引入重排，取决于业务对「精度 vs 延迟/成本」的权衡：

- **建议使用重排的场景**：
  - 问答类任务，答案是否命中对业务影响大（客服、法律、医疗、金融）。
  - 文档库以长文、专业领域文本为主，Bi-encoder 的粗粒度相似度不够用。
  - 多路召回（dense + sparse + 其他）后，需要一个统一的精排层融合排序。
  - 上下文窗口受限，必须把最相关的少数文档放在前面。

- **可以暂缓重排的场景**：
  - 文档库小、查询简单，Bi-encoder 的 top-K 已经足够准。
  - 对延迟极敏感（如实时对话），无法承受额外几百毫秒。
  - 成本敏感且 QPS 极高，重排开销难以承受。

一个经验法则：**当召回质量已经不错、但 top-K 里仍常混入「似相关而非答」的文档时，就是该上重排的信号**。

## 六、主流 reranker 模型

当前业界常用的重排模型大致分三类：

### 1. 开源可自部署

- **bge-reranker（智源）**：BAAI 出品，包括 reranker-base/large 及 v2/v2-gemma/v2-minicpm 等多语言版本。社区使用广泛，效果好，中文表现优秀，可本地 GPU/CPU 部署，是自建 RAG 的首选。
- **jina-reranker（Jina AI）**：jina-reranker-v2 等版本，多语言、支持长上下文，部署友好，质量在公开基准上名列前茅。
- **ms-marco-MiniLM 等学术模型**：基于 MS MARCO 训练的经典 Cross-encoder，适合快速实验，但泛化与多语言能力较弱。

### 2. 商业 API

- **Cohere Rerank**：最成熟的商业 reranker API 之一，提供 `rerank-english-v3.0`、`rerank-multilingual-v3.0` 等模型。调用方式简单：传入 query 和一组 documents，返回带分数的排序结果。支持上下文长度大、多语言，按 token 计费。适合不想自建 GPU、追求开箱即用的团队。
- **Voyage、Jina API 等**：同样提供托管的 rerank 服务，可作为 Cohere 的替代选项。

### 3. LLM-based reranker

- 用大模型本身（或专门微调的小模型）对每个候选做相关性判断，输出分数或排序。精度高但成本和延迟也最高，通常用于对最终上下文质量的极限优化，或在数据稀缺领域做少样本重排。

选型建议：自部署优先 bge-reranker（中文场景尤其推荐），快速验证或无 GPU 资源选 Cohere Rerank，对质量极致追求且预算充足可叠加 LLM-based reranker。

## 七、两阶段检索的参数关系：N >> K

重排的工程化形态是**两阶段检索**：

```
全库文档
  │
  ├─ 召回阶段（Bi-encoder + ANN）→ top-N （N 较大，如 50~200）
  │
  └─ 重排阶段（Cross-encoder）→ top-K （K 较小，如 3~10）
```

核心参数关系是 **N >> K**：

- **N（召回数量）**：决定重排的输入规模。N 越大，越有可能把真正相关的文档纳入候选，重排后精度上限越高；但 N 越大，重排延迟和成本也线性增长。典型取值 50~200。
- **K（最终保留数量）**：决定送入 LLM 的上下文条数。K 受 LLM 上下文窗口、token 预算和「lost in the middle」效应约束。典型取值 3~10。
- **N 与 K 的比例**：经验上 N 取 K 的 10~20 倍较均衡。例如最终想要 top-5，召回 top-50~100 再重排，往往能在精度与成本之间取得较好折中。

调参要点：

- N 太小：召回阶段漏掉了真正相关的文档，重排也救不回来——**召回率是天花板**。
- N 太大：边际收益递减，重排成本和延迟显著上升。
- K 太大：上下文噪声增加，LLM 反而容易「迷失在中间」，且推理成本上升。
- 在 N 固定时，提升 reranker 模型质量比单纯增大 K 更有效。

## 八、多路召回后重排的位置

现代 RAG 常采用**混合检索**：稠密向量（dense）召回语义相关，稀疏向量（sparse，如 BM25）召回关键词匹配，可能再加一路其他召回。多路结果分数尺度不一、来源不同，需要一个统一的精排层来融合排序，这正是重排的用武之地。

典型链路如下：

```
            ┌─ dense 召回 ─┐
query ──────┼─ sparse 召回 ┼─→ 合并去重 → top-N → Reranker → top-K → LLM
            └─ 其他召回 ────┘
```

重排在多路召回中的位置非常关键：

1. **去重与合并**：各路召回可能有重叠文档，先按 doc id 去重，再合并成统一的候选池（通常取并集或加权截断）。
2. **统一打分**：重排器对所有候选用同一模型、同一尺度重新打分，天然解决了多路分数不可比的问题，比简单的加权融合或 RRF 更精细。
3. **截断输出**：按重排分数取 top-K，送入 LLM 上下文组装。

注意：重排应放在**合并去重之后**，避免对同一文档重复打分浪费算力；也应放在**所有召回路完成之后**，确保候选池足够全面。

## 九、工程实践要点

- **批处理与并发**：对 N 个候选的打分请求应批量化送入模型，充分利用 GPU 并行；高 QPS 下可做异步重排或缓存（对相同 query+doc 对缓存分数）。
- **文档截断**：Cross-encoder 输入长度有限（如 512 或更长），超长 doc 需要做切分或截断，避免关键信息被丢弃或截断在后半段。
- **分数校准**：不同 reranker 输出的分数尺度不同，不要跨模型比较绝对分数，只比较同一 query 内的相对排序。
- **失败降级**：重排服务不可用时应降级到召回原始排序，保证链路可用。
- **评测**：用人工标注或 LLM-as-judge 对重排前后的 top-K 做对比，量化 reranker 带来的精度提升，再决定是否上线。

## 十、小结与使用建议

重排序是 RAG 检索链路中「用精度换延迟」的关键一环。它用 Cross-encoder 的联合编码弥补 Bi-encoder 粗粒度召回的不足，在多路召回之后、上下文组装之前，把候选池从 top-N 精排到 top-K，显著提升送入 LLM 的上下文质量。

使用建议：

1. **默认架构**：召回用 Bi-encoder + ANN（可选多路混合），重排用 Cross-encoder，是当前 RAG 的事实标准配置。
2. **参数关系**：牢记 N >> K，N 取 K 的 10~20 倍，先保召回率再谈精度。
3. **模型选型**：中文/自部署选 bge-reranker，开箱即用选 Cohere Rerank，极致质量叠加 LLM-based reranker。
4. **位置正确**：重排放在多路召回合并去重之后、LLM 上下文组装之前。
5. **成本意识**：重排是延迟和成本的主要来源，务必限制候选规模，并做批处理、缓存与降级。
6. **按需引入**：并非所有场景都必须重排，根据精度需求与延迟/成本预算权衡，用评测数据驱动决策。

一句话：**召回求广、重排求准**。Cross-encoder 让 RAG 在「快」与「准」之间找到了平衡，是大多数生产级 RAG 系统不可或缺的精排层。
