# 文本分块策略详解：固定、递归、语义与结构化分块

## 一、为什么要分块

在 RAG（Retrieval-Augmented Generation，检索增强生成）流水线中，文档摄入阶段需要把长文档切分为若干较小的、可检索单元，这一步称为**分块（Chunking）**。分块并不是可有可无的预处理工序，而是直接决定检索质量与生成质量的枢纽环节，其必要性源于三方面：

1. **嵌入模型的输入长度限制**。主流文本嵌入模型（如 BGE、E5、OpenAI text-embedding-3）通常有 512 或 8192 token 的输入上限，长文档无法一次性整体向量化。
2. **检索精度与上下文聚焦**。把整篇文档作为一个向量，语义会被过度平均稀释，导致召回模糊。分块使每个 chunk 聚焦于一个子主题，向量更"锐利"，检索更精准。
3. **生成成本与上下文窗口**。LLM 的上下文窗口有限且按 token 计费，向 prompt 注入大段无关文本既浪费成本又稀释信号，分块后只注入 Top-K 相关片段即可。

## 二、分块对检索的影响

分块策略通过三条链路影响最终效果：

- **召回率（Recall）**：chunk 过大或过小都会损害召回。过大导致语义混杂，相似度被稀释；过小则上下文残缺，关键信息被截断到相邻 chunk。
- **精确率（Precision）**：合理的边界使 chunk 自包含（self-contained），减少"答非所问"的噪声片段被召回。
- **生成质量**：送入 LLM 的片段若语义残缺或拼接无序，会诱导幻觉；分块策略直接影响上下文的连贯性与完整性。

一个常见的反直觉点是：**分块越细并不等于检索越好**。过细的 chunk 会丢失上下文，使检索到的片段无法独立支撑回答；过粗则削弱向量区分度。因此分块是精度与上下文完整性之间的权衡。

```
分块质量影响链：
  分块策略 → chunk 边界/大小 → 向量表示质量 → 召回与排序 → 注入上下文 → 生成答案质量
```

## 三、固定大小分块

最朴素也最常用的策略：按固定字符数或 token 数切分，并保留一定**重叠（overlap）**以防语义被硬切断。

- **按字符**：实现简单、速度快、无依赖，适合纯文本或快速基线。
- **按 token**：用分词器（如 tiktoken）按 token 计数，与嵌入模型口径一致，更可控。

关键参数为 `chunk_size`（块大小）与 `chunk_overlap`（重叠量，通常取 10%~20%）。重叠可保证句尾与句首在相邻 chunk 中各出现一次，缓解边界信息丢失。

**优点**：实现成本极低、可复现、吞吐高、参数少易调优。
**缺点**：完全无视语义与文档结构，可能在句中或词中断开，产生语义残缺的 chunk；对表格、代码、列表等结构化内容破坏严重。
**适用场景**：日志、新闻正文、质量要求中等的通用知识库，或作为离线批量处理的快速基线。

## 四、递归字符分块

LangChain 的 `RecursiveCharacterTextSplitter` 是工业界事实标准。其核心思想是**按分隔符优先级递归切分**：给定一组按优先级排序的分隔符，先尝试用最高优先级分隔符切分；若切出的块仍超过 `chunk_size`，再退到下一级分隔符，直到无法继续细分或满足大小约束。

典型的分隔符优先级为：

```
["\n\n", "\n", "。", " ", ""]
段落 > 换行 > 句号 > 空格 > 字符
```

切分后再把相邻小块合并，直到接近 `chunk_size`，并用 `chunk_overlap` 控制重叠。这种"先结构后字面"的层级回退，使分块在绝大多数情况下沿段落、句子边界进行，仅在不得已时才侵入句内。

```
递归分块流程示意：
  原始文本
    │  尝试 "\n\n" 切分
    ▼
  段落级 chunk（仍超限？）
    │  是 → 退到 "\n" 切分
    ▼
  行级 chunk（仍超限？）
    │  是 → 退到 "。" 切分
    ▼
  句级 chunk → 合并至 chunk_size → 加 overlap → 输出
```

**优点**：实现成熟、参数化好、对多数自然语言文本边界保持良好、吞吐可观；分隔符列表可定制以适配中文（如加入 `。！？\n`）或代码（如 `class `、`def `、`\n\n`）。
**缺点**：分隔符仍是语法启发式而非语义判断，遇到长无标点段落或语义跨段落主题时仍会切偏；对 Markdown/HTML 等带层级结构的文档感知不足。
**适用场景**：绝大多数通用 RAG 项目的默认首选，尤其适合混合内容的文档库。

## 五、语义分块

语义分块（Semantic Chunking）不再依赖固定窗口或语法分隔符，而是**根据语义连贯性动态切分**。典型做法（Greg Kamradt 的方案）如下：

1. 将文本切分为细粒度句子；
2. 为每句生成嵌入向量；
3. 计算相邻句子向量的余弦相似度；
4. 当相邻相似度低于某阈值（如百分位阈值）时，判定语义发生跳变，在此处插入分块边界。

```
语义分块示意：
  句1 ─ 句2 ─ 句3 │ 句4 ─ 句5 │ 句6 ─ 句7 ─ 句8
  (相似度连续高)   (跳变)    (连续)   (跳变)
         ↓           ↓
       chunk A    chunk B   chunk C
```

**优点**：边界贴合真实语义切换，chunk 内部主题更聚焦，召回精度通常优于固定/递归策略；适合长篇叙述、对话、课程逐字稿。
**缺点**：需对所有句子批量调用嵌入模型，计算与延迟显著上升；阈值与窗口参数对结果敏感、难调；对短句密集文本效果退化；离线可用，实时摄入成本高。
**适用场景**：对召回质量要求高、文档以叙述性长文为主、且预算允许离线嵌入的场景。

## 六、文档结构感知分块

这类策略利用文档本身的**结构信号**（标题、段落、列表、表格、Markdown 标题层级、HTML 标签）来决定切分边界，使每个 chunk 尽量对应一个结构完整的语义单元。

- **MarkdownHeaderTextSplitter**：按 Markdown 标题层级（`#`、`##`、`###`）切分，并把所属标题路径作为元数据附加到每个 chunk，便于上下文溯源与过滤检索。
- **HTMLHeaderTextSplitter**：按 `<h1>`~`<h6>`、`<section>` 等标签切分，保留 DOM 上下文。
- **按段落/表格/代码块感知**：借助文档解析器（如 Unstructured、LayoutParser）识别版式元素，对表格、代码块整块保留或特殊处理，避免被字符切分打碎。

```
结构化分块示意（Markdown）：
  # 第一章 概述          ── H1 元数据
    ## 1.1 背景          ── H2 元数据
      正文段落 A         → chunk(归属 H1/H2)
      正文段落 B         → chunk(归属 H1/H2)
    ## 1.2 目标          ── H2 元数据
      正文段落 C         → chunk(归属 H1/1.2)
```

**优点**：chunk 与文档逻辑结构对齐，自包含性强，元数据可显著增强过滤检索与可解释性；对表格、代码、条款类内容保护到位。
**缺点**：依赖文档解析质量与版式规整度；标题层级过深或文档无明确结构时退化；实现相对复杂。
**适用场景**：技术文档、产品手册、法律合同、API 文档、Markdown/HTML 知识库等结构化程度高的语料。

## 七、父子分块 / Small-to-Big

上述策略在"检索单元"与"生成单元"上常常是同一个 chunk，而父子分块解耦了二者：

- **检索用小块（Parent→Child 子块）**：小 chunk 向量更聚焦、召回更准。
- **生成用大块（Parent）**：召回小 chunk 后，将其所属的父 chunk（更完整上下文）送入 LLM 生成。

常见实现有 `ParentDocumentRetriever`（LangChain）与 **Small-to-Big / Sentence-Window Retrieval**（检索句子，返回其前后窗口或所属段落）。

```
父子分块示意：
  父块（大，用于生成） ┌─ 子块1（小，向量化+检索命中）
                      ├─ 子块2
                      └─ 子块3
  命中子块 → 取回父块 → 送入 LLM
```

**优点**：兼得检索精度与生成上下文完整性，显著缓解"chunk 太小导致上下文断裂"的老问题。
**缺点**：存储与索引成本上升（需维护父子映射、双重索引）；实现与运维复杂度增加。
**适用场景**：对生成质量要求高、愿为精度付出存储与工程成本的进阶 RAG 系统。

## 八、策略对比

| 策略 | 切分依据 | 实现成本 | 上下文完整性 | 召回精度 | 典型适用场景 |
|---|---|---|---|---|---|
| 固定大小分块 | 字符/token 数 | 低 | 弱（易断句） | 中 | 日志、通用文本、快速基线 |
| 递归字符分块 | 分隔符优先级 | 低 | 中 | 中高 | 通用 RAG 默认首选 |
| 语义分块 | 嵌入相似度跳变 | 高 | 高 | 高 | 叙述性长文、对话、课程稿 |
| 结构感知分块 | 标题/段落/版式 | 中 | 高 | 高 | 技术文档、手册、合同 |
| 父子/Small-to-Big | 检索-生成解耦 | 高 | 很高 | 高 | 高质量生成型 RAG |

## 九、分块质量评估

分块策略需要可量化评估，常见维度与指标：

- **检索指标**：在标注问答集上测 Recall@K、MRR、NDCG，对比不同策略的召回与排序能力。
- **生成指标**：用 LLM-as-a-Judge 或人工评估答案的忠实度（Faithfulness）与相关性，观察分块对幻觉率的影响。
- **块内聚度**：统计 chunk 内部主题一致性，如用嵌入向量计算 chunk 内句子平均相似度，过低说明语义混杂。
- **块间边界合理性**：人工抽检 chunk 是否在语义完整处切断，是否截断表格/代码/定义。
- **大小分布**：监控 chunk 长度的均值、方差与极值，过短（<50 token）或过长（>1500 token）占比过高需调参。

实践建议先以递归字符分块建立基线，再用检索指标做 A/B 对比，逐步引入语义或结构感知策略，并通过父子分块补齐生成上下文。

## 十、小结与选型建议

分块没有银弹，选型应遵循"**先结构、再语义、后大小**"的原则：

1. **结构化文档优先结构感知分块**：Markdown/HTML/手册类语料，优先 `Markdown/HTMLHeaderTextSplitter`，保留层级元数据。
2. **通用文本默认递归字符分块**：作为低成本、高性价比的默认基线，再按需叠加语义或父子策略。
3. **叙述性长文用语义分块**：课程、对话、长篇说明文，预算允许时显著提升召回精度。
4. **追求生成质量用父子分块**：检索小 chunk、生成用父块，解耦精度与上下文。
5. **固定大小分块用于基线与日志**：作为快速对照或对结构不敏感的海量文本。

最终，分块策略应与嵌入模型、检索器、重排器协同调优，并以端到端检索与生成指标为准绳持续迭代。
