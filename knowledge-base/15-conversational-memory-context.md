# 多轮对话记忆与上下文管理

## 一、为什么多轮对话比单轮难

单轮问答（QA）中，用户的 query 自包含，检索与生成都围绕该 query 展开。而真实的多轮对话（如客服、编程助手、个人助理）中，当前轮的 query 往往是"残缺"的——它依赖前文才能被正确理解。这带来三类核心挑战：

1. **指代消解（Coreference Resolution）**：用户说"它的价格是多少"，"它"指代上一轮提到的某个实体。若不做消解，检索系统会用"它的价格是多少"去查向量库，几乎不可能命中。
2. **省略恢复（Ellipsis Recovery）**：用户接着问"那有红色的吗"，省略了主语。需要从前文补全为"[某商品]有红色的吗"。
3. **话题漂移（Topic Drift）**：对话可能在多个话题间跳跃，前 3 轮聊 A 产品，第 4 轮回到 A，第 5 轮又问 B。若把全部历史无差别塞入上下文，既浪费 token 又可能引入噪声，导致模型被无关历史带偏。

此外，多轮对话还要解决一个工程问题：**上下文窗口有限**。即使 200K 窗口的模型，也无法把一个持续数小时、上百轮的对话完整保留——成本、延迟、注意力稀释都会成为瓶颈。因此，"记什么、忘什么、怎么压缩"成为对话系统的核心设计问题。

## 二、对话状态与会话历史管理

### 2.1 对话状态（Dialog State）

对话状态是对"当前对话进展到哪里、关键事实是什么"的结构化抽象，区别于原始的消息历史。它可以包含：

- **槽位（Slots）**：任务型对话中的关键参数，如订机票场景的"出发地/目的地/日期/舱位"。
- **已确认事实**：用户已经表达过偏好（如"我预算 5000 以内"），后续轮次需遵守。
- **当前话题焦点**：最近 N 轮讨论的实体或意图，用于消解指代。
- **未决问题**：系统向用户提问后尚未得到回答的开放项。

状态可以用结构化 JSON、键值存储或小型状态机维护。好处是：生成回复时可以直接注入结构化状态，而不必让 LLM 从冗长历史里"再推理一次"，既省 token 又更稳定。

### 2.2 会话历史管理

会话历史通常按"会话（session）—轮次（turn）—消息（message）"三级组织：

- **Session**：一个连续对话单元，有唯一 session_id，对应一个用户的一次交互过程。
- **Turn**：一次"用户说—系统答"的往返，是记忆的最小调度单位。
- **Message**：单条消息（user/assistant/tool），携带角色、内容、时间戳、token 数等元信息。

工程上需要为每条消息记录 token 估算值，以便后续做窗口裁剪时无需重新计算。同时建议为每轮打上"话题标签"或"意图标签"，为后续的摘要分段、检索筛选提供依据。

## 三、记忆窗口策略

如何从会话历史中选取要送入 LLM 的内容，是记忆策略的核心。常见策略如下表对比：

| 策略 | 机制 | 优点 | 缺点 | 适用场景 |
|---|---|---|---|---|
| **全量记忆** | 保留所有历史消息原文 | 信息无损，指代/省略可完整还原 | token 爆炸，成本/延迟高，长对话注意力稀释 | 短对话（<20 轮）、调试期 |
| **滑动窗口** | 只保留最近 K 轮原文 | 实现简单，延迟可控 | 丢失早期关键事实（如用户预算、身份） | 闲聊、轻量助手 |
| **摘要记忆** | 对超出窗口的历史做 LLM 摘要，保留摘要+近期原文 | 兼顾长程信息与 token 预算 | 摘要可能丢细节，有额外 LLM 调用成本 | 长对话、客服、Agent |
| **摘要+滑窗混合** | 近期原文 + 中期滚动摘要 + 远期关键事实 | 平衡性最好 | 实现复杂度高 | 生产级长对话系统 |
| **向量记忆检索** | 把历史切分为 chunk 入向量库，按当前 query 检索相关片段 | 可扩展到超长会话/跨会话 | 检索召回不稳定，指代依赖检索质量 | 跨会话长期记忆、Agent 记忆层 |

实践中最常用的是"摘要+滑窗混合"：最近 K 轮保留原文以保证指代可消解，更早的历史压缩为递进式摘要（summary），最早的关键事实抽成结构化"用户画像/会话事实表"单独存放。

## 四、历史压缩：用 LLM 摘要历史

当历史超出预算时，需要对旧历史进行压缩。最主流的做法是 **LLM 摘要**：

1. **触发条件**：当累计 token 超过阈值（如窗口的 60%）时触发。
2. **摘要范围**：选取窗口外最旧的一段（如第 1–N 轮），而非全量重摘，避免重复劳动。
3. **增量更新**：维护一个 rolling summary，每次新增待压缩段时，把"旧 summary + 新压缩段"合并生成新 summary。这样 summary 始终是单条、最新、长度可控的。
4. **摘要 prompt 要点**：
   - 明确要求保留"事实、偏好、约束、未决问题"，丢弃寒暄与冗余。
   - 保留实体名与关键数值，便于后续检索与指代。
   - 指明输出格式（如分点列出），便于注入 prompt。

摘要的代价是信息损失。为缓解，可对摘要内容做"关键事实抽取"——把"用户预算 5000""用户是 Java 开发者"这类硬事实单独存入结构化字段，摘要只负责叙事性脉络。这样即便摘要丢失细节，关键约束依然在。

## 五、指代消解与上下文相关 Query 改写

这是多轮对话与 RAG 结合时的关键环节。当用户发出依赖前文的 query 时，直接用它检索会失败。标准做法是 **基于对话历史的 query 改写（context-dependent query rewriting）**：

- **独立 query（Standalone Query）**：用一个轻量 LLM，结合最近若干轮历史，把当前 query 改写为"自包含"形式。例如历史"我想买笔记本" + 当前"它的价格" → 改写为"ThinkPad X1 笔记本的价格"。
- **HyDE/假设文档**：在指代严重时，可让 LLM 先生成一段"假设性回答"作为检索 query，再检索。
- **多 query 生成**：对模糊 query 生成多个改写版本并行检索，再合并去重，提升召回。

改写后的 query 一路用于检索，另一路与原 query 一起作为生成时的上下文。注意：改写 query 用于检索，但**生成时建议保留用户原始 query**，以保持回答口吻贴近用户意图。

指代消解也可在 prompt 层面隐式完成——只要把足够的历史塞给生成 LLM，它通常能自行消解。但隐式消解依赖窗口内有足够历史，且不可控；显式改写则更可靠、可调试、可缓存，是生产系统的首选。

## 六、记忆与检索的交互：用历史生成检索 Query

在对话式 RAG 中，检索不是"拿当前 query 去查"这么简单。需要用对话历史反哺检索：

1. **检索 query 构建**：融合"当前轮 + 最近历史 + 会话状态槽位"，生成检索 query。例如槽位"目标产品=ThinkPad X1"可直接拼入 query，提升命中。
2. **历史感知检索**：对历史中提到的实体做实体链接，将其作为检索的过滤条件或加权项。
3. **多路检索**：一路用改写 query 检索知识库，一路用原 query 检索"对话记忆库"（见下节），两路结果合并。
4. **去重与重排**：检索回来的片段需经重排（rerank），并结合"话题焦点"过滤掉与当前话题无关的历史片段。

一个常见误区是只检索外部知识库，忽略了"对话自身历史"也是重要的检索源。长期记忆库的检索往往和知识库检索同等重要。

## 七、长期记忆 vs 短期记忆

借鉴认知科学，对话记忆可分为两层：

- **短期记忆（Working Memory）**：当前会话内的上下文，存在于 LLM 上下文窗口中，包含最近轮次原文、当前摘要、当前检索片段。生命周期为一个 session，会话结束即释放。
- **长期记忆（Long-term Memory）**：跨会话持久化的记忆，存储在向量库或结构化数据库中。包括用户画像、历史偏好、过往决策、事实记忆等。当新会话开始时，按用户身份检索相关长期记忆注入上下文。

长期记忆的写入需要谨慎——不是所有历史都值得长期保存。常见策略是"重要度过滤"：用一个 LLM 判断某轮对话是否包含值得长期记住的事实（用户偏好、关键决策、身份信息），仅对高重要度内容写入长期记忆，并打上时间戳、话题、来源会话等元数据以便后续检索与衰减。

长期记忆的检索通常是"按当前 query + 用户 ID"做向量检索，返回 top-K 后注入 prompt 的"长期记忆"区块。需注意控制返回数量，避免噪声淹没当前任务。

## 八、上下文窗口预算分配

一个生产级对话 prompt 通常要同时容纳：系统指令、用户画像/长期记忆、会话摘要、近期原文历史、当前检索上下文、当前用户 query、输出格式约束。窗口是有限的，必须做预算分配。典型分配（以 128K 窗口为例）：

- 系统指令与角色设定：约 5–10%
- 长期记忆/用户画像：约 5–10%
- 会话摘要：约 5–10%
- 近期原文历史（最近 K 轮）：约 15–25%
- 检索上下文（RAG 片段）：约 30–40%
- 当前 query 与输出约束：约 5%
- 输出预留（生成空间）：约 20–30%

关键取舍原则：

1. **检索上下文 vs 历史**：当检索片段多时，应主动压缩历史（更激进地摘要）；当历史中含关键约束时，应减少检索 top-K。两者是此消彼长的预算博弈。
2. **近期原文不可省**：为保证指代消解，最近 2–3 轮原文必须保留，即使检索上下文很满。
3. **输出预留要足**：预留不足会导致回答被截断，尤其在长篇生成（代码、报告）场景。
4. **分级淘汰**：超预算时按"远期原文 → 检索片段 top-K 尾部 → 摘要细节"顺序依次裁剪，保留核心。

建议在系统中实现一个"上下文组装器"组件，统一管理各类内容注入、token 计数与裁剪，而不是把逻辑散落在各处。

## 九、记忆持久化

记忆最终要落到存储：

- **短期记忆**：可存于 Redis 等内存数据库，按 session_id 索引，设 TTL（如 24 小时无活动自动过期）。
- **长期记忆**：向量库（如 FAISS/Milvus）存语义记忆，结构化数据库（PostgreSQL）存事实表与用户画像。
- **会话日志**：原始对话全量落盘到对象存储或日志系统，用于审计、回放、离线分析、训练数据生产。
- **版本化**：摘要与状态变更建议保留版本，便于排查"为什么会话走到这一步"以及回滚。

持久化设计要考虑一致性：摘要更新与状态写入应在同一事务或采用幂等写，避免重启后状态不一致。跨实例部署时，会话需可路由到持有该 session 状态的节点，或采用共享存储。

## 十、小结与策略建议

多轮对话记忆的本质，是在有限窗口内最大化"对当前轮有用的信息密度"。核心建议：

1. **分层记忆**：近期原文 + 中期摘要 + 远期结构化事实 + 跨会话长期记忆，各司其职。
2. **显式改写优先**：用历史改写 query 做检索与消解，比纯靠 LLM 隐式推理更可控、可缓存。
3. **摘要要保事实**：摘要 prompt 强调保留实体、数值、约束，并辅以结构化事实抽取兜底。
4. **预算先行**：用统一的上下文组装器管理 token 预算，明确各类内容的配额与裁剪优先级。
5. **检索与记忆一体**：把对话历史作为一路检索源，与知识库检索并列、合并、重排。
6. **持久化与可观测**：全量日志落盘，状态变更版本化，便于回放与排查。

记住一条原则：**记忆不是越多越好，而是越准越好**。优秀的对话系统不是把所有历史塞给模型，而是精准地把"当前轮最需要的那一小撮上下文"送到模型眼前。这是区分玩具 Demo 与生产级对话系统的分水岭。
