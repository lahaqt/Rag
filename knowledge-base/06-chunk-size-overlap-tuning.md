# Chunk 大小与重叠调优：召回率与上下文窗口的平衡

在 RAG 系统中，分块（Chunking）是连接「原始文档」与「检索 / 生成」的咽喉环节。Chunk 的大小与重叠（Overlap）并非可以随意拍板的工程参数，而是直接决定三件事的杠杆：嵌入向量的语义完整度、检索阶段的召回与精确率、以及最终送入 LLM 上下文窗口的预算分配。本文围绕这三个维度，给出可复用的调优方法论。

## 一、为什么 Chunk 大小是一个双向杠杆

很多团队的直觉是「Chunk 越大，上下文越完整，效果越好」。这只说对了一半。Chunk 大小对召回率、精确率和上下文占用的影响是**双向且相互制约**的，不存在一个全局最优值，只存在针对特定语料、Embedding 模型与 LLM 的局部最优。

### 1.1 Chunk 过大的代价

当一个 Chunk 过大（例如超过 1024 tokens 甚至逼近 2048）时，会同时压制召回与精确率：

- **语义稀释（Semantic Dilution）**：Embedding 模型本质上是在对一段文本做「语义质心」压缩。Chunk 越长，内部主题越杂，向量越接近多个语义的加权平均。结果是该 Chunk 与任一具体 query 的相似度都被拉低，既可能在 top-K 之外（召回下降），也可能因为混入无关内容而稀释了相关性信号（精确率下降）。
- **召回粒度变粗**：检索以 Chunk 为最小可返回单位。一个 1500 tokens 的 Chunk 即使只包含 200 tokens 的真正相关内容，也会整块返回，剩下的 1300 tokens 是噪声，直接挤占上下文窗口。
- **Embedding 输入截断风险**：多数主流 Embedding 模型（如 `bge-large`、`text-embedding-3-large`、`e5-mistral`）的最大输入在 512～8192 tokens 之间。一旦 Chunk 超过模型上限，要么被截断（丢失尾部语义），要么被强行拆分（破坏原始分块意图）。

### 1.2 Chunk 过小的代价

反过来，Chunk 过小（例如仅 64～128 tokens）同样有害：

- **语义碎片化**：一个完整的论断被切成若干残句，单个 Chunk 无法独立表意。检索命中的可能是半句话，LLM 拿到的也是断章取义的片段，生成质量下降。
- **数量爆炸，检索噪声上升**：Chunk 越小，知识库总块数越多。在固定 top-K 下，召回的候选更多但单块信息量更少，精确率下降；同时向量库存储与检索计算成本线性增长。
- **上下文拼接成本**：小 Chunk 往往需要靠 Overlap 或父子块（Parent-Child）策略拼回完整语义，工程复杂度上升。

### 1.3 参数影响对照表

下表汇总了 Chunk 大小在关键指标上的双向影响，便于快速定位问题方向：

| 维度 | Chunk 偏小（如 128） | Chunk 适中（如 512） | Chunk 偏大（如 1536） |
|---|---|---|---|
| 单块语义完整度 | 差（残句） | 良好 | 好，但主题混杂 |
| Embedding 语义聚焦度 | 高 | 中 | 低（语义稀释） |
| 召回率（Recall@K） | 偏低（碎片难命中） | 较高 | 偏低（相似度被稀释） |
| 精确率（Precision@K） | 偏低（信息量不足） | 较高 | 偏低（噪声占比高） |
| 上下文 token 占用 | 低 | 中 | 高，易触顶 |
| 知识库总块数 | 多 | 中 | 少 |
| 检索/存储成本 | 高 | 中 | 低 |

经验上，**256～1024 tokens** 是多数通用语料的甜区，512 是一个稳妥的起点。

## 二、Overlap 的作用与代价

Overlap（重叠）指相邻 Chunk 之间共享的一段文本，通常取 Chunk 大小的 10%～25%，常见值为 50～200 tokens。

### 2.1 Overlap 解决的真问题

Overlap 的核心价值在于**跨越边界的语义连续性**。当一个完整论断恰好被切分点拆开时：

- 无 Overlap：关键信息被一分为二，任一单独 Chunk 都不完整，检索可能只命中一半，生成端拿到的也是断裂上下文。
- 有 Overlap：被切分的内容会在相邻 Chunk 中各出现一次，提升边界信息被完整召回的概率。

此外，Overlap 对**多模态锚点**（如表格、代码块、标题-正文对）有保护作用——避免一个 Chunk 的开头恰好丢失了上一段的关键定义或上文。

### 2.2 Overlap 的代价

Overlap 不是免费的：

- **存储与索引膨胀**：重叠部分被重复嵌入并入库，向量库体积按 `1 + overlap/chunk_size` 比例增长。例如 512 tokens 配 128 overlap，存储约增加 25%。
- **召回去重负担**：同一段文本出现在多个 Chunk 中，检索 top-K 时可能返回高度重复的多个块，挤掉本应出现的其他相关块，变相降低 Recall 的有效覆盖。这通常需要在检索后做去重或 MMR（Maximal Marginal Relevance）重排。
- **边际收益递减**：当 Overlap 超过约 25% 后，对边界召回的提升趋平，而存储与去重成本继续上升，性价比下降。

经验上，Overlap 取 Chunk 大小的 **10%～20%** 即可覆盖绝大多数边界连续性问题。

## 三、与 Embedding 模型最大输入的对齐

Chunk 大小的硬上限由 Embedding 模型的最大输入决定，这是常被忽略的工程约束：

- `text-embedding-3-small / large`（OpenAI）：8191 tokens
- `bge-large-zh-v1.5`：512 tokens
- `bge-m3`：8192 tokens
- `e5-mistral-7b-instruct`：4096 tokens
- `gte-large-zh`：512 tokens

**对齐原则**：Chunk 大小应显著小于模型上限，留出 tokenizer 与分词差异的余量，避免触发截断。例如使用 `bge-large`（512 上限）时，Chunk 取 256～480 较为安全；使用 `bge-m3` 或 `text-embedding-3` 时才有空间把 Chunk 提到 768～1024。

此外要注意模型对**最优输入长度**的偏好：部分模型在 128～512 tokens 区间表现最佳，过长输入即使不截断，向量质量也会下降。建议以模型官方评测报告的「sweet spot」为参考，而非简单贴着上限走。

## 四、与 LLM 上下文窗口的预算分配

Chunk 大小与数量共同决定送入 LLM 的上下文结构。给定一个上下文窗口 `C`（如 32K、128K、200K），需要在这些消费者之间分配预算：

1. **系统提示 + Few-shot 示例**：固定开销，通常 1K～4K。
2. **用户 query 与指令**：通常 0.2K～2K。
3. **检索召回的 Chunk**：`top_K × chunk_size`，是最大的可变项。
4. **生成输出预留**：需为 LLM 回答预留 `max_output_tokens`，不能被检索内容挤占。

**预算公式**（粗略）：

```
可用检索预算 ≈ C - 系统提示 - query - 预留输出
top_K × effective_chunk_size ≤ 可用检索预算
```

其中 `effective_chunk_size` 应计入 Overlap 带来的实际平均长度。实践建议：

- **小窗口模型（8K～32K）**：优先小 Chunk（256～512）+ 中等 top-K，靠数量换覆盖。
- **大窗口模型（128K+）**：可放宽到中大 Chunk（512～1024）+ 较大 top-K，但需警惕「长上下文迷失」（Lost in the Middle）——LLM 对中段信息的利用率下降，因此不要无脑堆满窗口，**召回精筛比堆量更重要**。
- **混合策略**：用小 Chunk 做召回命中，再用 Parent-Child 或滑动窗口回溯其所属的大段落送入 LLM，兼顾检索精度与生成上下文完整度。

## 五、Chunk 大小 vs 数量的权衡

这是一个常被混淆的维度。在固定上下文预算下，「大 Chunk + 小 K」与「小 Chunk + 大 K」可以达到相近的总 token 量，但行为差异显著：

| 策略 | 检索行为 | 生成行为 | 适用场景 |
|---|---|---|---|
| 大 Chunk + 小 K | 单块信息密集，噪声也密集 | 上下文连续，但可能含无关段落 | 长文问答、需要完整段落的任务 |
| 小 Chunk + 大 K | 覆盖面广，单块聚焦 | 上下文分散，需 LLM 自行整合 | 多点事实聚合、FAQ 检索 |

权衡要点：

- 大 Chunk 的优势在**生成端**的上下文完整性，劣势在**检索端**的语义稀释。
- 小 Chunk 的优势在**检索端**的聚焦，劣势在**生成端**的碎片化。
- 若检索质量（Recall@K）是瓶颈，优先缩小 Chunk；若生成质量（答案完整度）是瓶颈，优先放大 Chunk 或引入 Parent-Child 回溯。

## 六、评估驱动的调参方法

Chunk 大小与 Overlap 没有放之四海皆准的值，**必须由评估指标反向驱动**。推荐流程如下：

### 6.1 建立评测集

构建一个覆盖典型 query 分布的 golden set，每条 query 标注：

- 期望命中的文档 / 段落（用于计算 Recall@K）。
- 期望答案（用于计算生成端指标）。

### 6.2 扫参实验

以 Chunk 大小为主变量、Overlap 为副变量做网格扫描，常用组合：

- Chunk：`[256, 384, 512, 768, 1024]`
- Overlap：`[0, 50, 100, 150]`（或按 Chunk 的 10%、20% 取值）

对每组参数计算：

- **Recall@K**：top-K 内是否包含期望段落，反映召回能力。
- **Precision@K / nDCG@K**：召回块的相关性密度，反映噪声水平。
- **MRR**：期望段落的排名位置，反映排序质量。
- **端到端答案指标**：Faithfulness、Answer Relevance、正确率。

### 6.3 反馈解读

- 若 Recall@K 随 Chunk 增大先升后降，拐点即为该语料的语义完整度临界值，是 Chunk 上限的参考。
- 若 Recall 不足但 Precision 尚可，优先**增大 top-K 或减小 Chunk**，扩大覆盖。
- 若 Recall 够但答案质量差，问题在**生成端**，应增大 Chunk 或引入上下文回溯。
- 若增大 Overlap 后 Recall 明显上升，说明原 Chunk 边界切分损伤严重，可适当提升 Overlap；若提升不明显，则维持低值以省存储。

### 6.4 与重排协同

Chunk 调优应与重排（Rerank）协同：先用较小 Chunk + 较大 top-K 做粗召回保 Recall，再用 Cross-Encoder 重排筛 top-N 送入 LLM。这种「宽进严出」的方案通常比单纯调大 Chunk 更高效。

## 七、不同内容类型的差异化取值

Chunk 策略必须匹配内容形态，一刀切是常见反模式：

### 7.1 FAQ / 问答对 / 结构化条目

- **取值**：单条问答即一个 Chunk，通常 64～256 tokens。
- **策略**：按条目边界分块，**Overlap 设为 0**（条目本身自洽，无需重叠）。
- **要点**：保留「问题 + 答案」整体，避免拆分；可为每个条目附加元数据（标题、类目）提升过滤检索。

### 7.2 长文 / 文档 / 手册

- **取值**：512～1024 tokens，Overlap 50～150。
- **策略**：优先按结构（标题、段落、小节）分块，而非纯按 token 数硬切；结构边界优先，token 上限兜底。
- **要点**：保留每个 Chunk 的标题上下文（如「section 标题 + 内容」），避免正文 Chunk 丢失所属章节语义。

### 7.3 代码 / 技术文档

- **取值**：以函数 / 类 / 方法为粒度，通常 200～800 tokens。
- **策略**：按 AST 节点分块，**Overlap 设为 0** 或极小；绝不把函数从中间截断。
- **要点**：保留函数签名与注释；跨文件依赖可在元数据中标注 import 关系，便于检索时还原上下文。

### 7.4 表格 / 半结构化数据

- **取值**：整表或按行分块，视表大小而定。
- **策略**：小表整块入库并附表头描述；大表按行/行组分块但**每块重复携带表头**，相当于一种「结构化 Overlap」。
- **要点**：为表格生成自然语言摘要并一同嵌入，提升与自然语言 query 的匹配度。

## 八、小结与默认参数建议

Chunk 大小与 Overlap 的调优本质是在**语义完整度、检索聚焦度、上下文预算**三者间寻找平衡，核心结论如下：

- **Chunk 大小是双向杠杆**：过小则碎片化、召回不足；过大则语义稀释、噪声挤占上下文。通用甜区在 256～1024 tokens。
- **Overlap 以小为美**：解决边界连续性问题即可，10%～20% 覆盖大多数场景，超过 25% 性价比骤降。
- **对齐 Embedding 模型上限**：Chunk 应显著小于模型最大输入，并参考模型的最优输入区间，而非贴上限走。
- **上下文预算优先分配**：先扣除系统提示、query、输出预留，再决定 `top_K × chunk_size`，大窗口模型也要警惕长上下文迷失。
- **评估驱动**：用 Recall@K、Precision@K、MRR 与端到端答案指标做网格扫描，由拐点而非直觉定参。
- **内容差异化**：FAQ 按条目、长文按结构、代码按 AST、表格带表头，切忌一刀切。

**默认参数建议**（作为未做评测时的起点，随后用评估集校准）：

| 内容类型 | Chunk 大小（tokens） | Overlap | 备注 |
|---|---|---|---|
| 通用长文 | 512 | 64～100 | 起点配置，最常用 |
| FAQ / 问答条目 | 单条（64～256） | 0 | 按条目边界 |
| 代码 / 技术文档 | 按函数/类（200～800） | 0 | AST 分块 |
| 表格 | 整表或带表头行组 | 表头重复 | 附自然语言摘要 |
| 小窗口 LLM 场景 | 256～384 | 32～64 | 控制上下文占用 |
| 大窗口 LLM 场景 | 512～1024 | 100～150 | 配合重排精筛 |

一句话原则：**从 512 / 64 起步，用 Recall@K 驱动校准，按内容类型差异化收尾**。参数本身不是终点，可观测、可评估、可迭代的调参闭环才是。
