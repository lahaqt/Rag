# RAG 概念总览：检索增强生成的原理、动机与对比

## 一、什么是 RAG

RAG（Retrieval-Augmented Generation，检索增强生成）是一种将**外部知识检索**与**大语言模型（LLM）生成能力**相结合的技术范式。它的核心思想非常朴素：在模型回答用户问题之前，先从一个外部知识库（文档、数据库、向量索引等）中检索出与问题最相关的若干片段，再把这些片段作为上下文连同问题一起交给 LLM，由 LLM 基于这些"参考资料"生成最终答案。

用一个生活中的类比来理解：传统的 LLM 就像一个只能凭记忆考试的学生，他读过的书都装在脑子里，但考场上不能翻书；而 RAG 则像一场**开卷考试**，学生遇到问题时可以先去查资料，再结合自己的理解作答。显然，开卷考试在需要准确事实、最新信息和引用来源的场景下更可靠。

RAG 最初由 Lewis 等人在 2020 年的论文《Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks》中系统提出，最初用于解决开放域问答等"知识密集型"任务。随着大模型的兴起，它迅速成为企业落地大模型应用的事实标准方案之一。

可以用一个简化的公式描述 RAG 的工作方式：

$$
\text{Answer} = \text{LLM}(\text{Question} \oplus \text{Retrieve}(\text{Question}, \text{KnowledgeBase}))
$$

其中 $\oplus$ 表示将检索到的上下文拼接到用户问题中，再整体输入给 LLM。本质上，RAG 没有改变 LLM 本身的参数，而是通过**动态注入外部知识**来增强其表现。

## 二、为什么需要 RAG：LLM 的固有局限

要理解 RAG 的动机，必须先理解单纯使用 LLM 时面临的几类痛点。这些痛点共同推动了 RAG 的诞生与流行。

### 1. 知识截断（Knowledge Cutoff）

LLM 的知识来源于训练数据，而训练有一个截止时间。模型一旦训练完成，对之后发生的世界事件、产品发布、法规变更等就一无所知。例如一个 2023 年训练完成的模型无法回答"2025 年某新法规的具体条款"。重新训练或继续预训练成本极高且周期长，无法做到日更甚至小时更。

### 2. 幻觉问题（Hallucination）

LLM 是基于概率生成下一个 token 的模型，当它对某个事实不确定时，并不会"闭嘴"，而是倾向于生成看似合理却实际错误的回答，这种现象被称为幻觉。幻觉在事实性问答、医疗法律等专业领域是致命缺陷。RAG 通过把权威检索片段作为依据，显著降低了模型"编造"事实的倾向，并使答案可追溯。

### 3. 领域与私有数据问题

通用大模型在公开语料上训练，对企业内部的文档、产品手册、客户数据、内部知识库等私有信息一无所知。这些数据既不能（涉及隐私与合规）也不必送进模型预训练。RAG 让企业可以在不泄露、不重训的前提下，把私有知识"外挂"给模型使用。

### 4. 可更新性与成本

微调或继续预训练需要数据、算力和时间，更新一次知识就要重训一遍，性价比低。RAG 的知识库是独立的，更新知识只需增删改文档并重建索引，几乎是实时的，且成本与模型训练相比低得多。

### 5. 可解释与可溯源

在很多场景下，光给出答案不够，还需要告诉用户"这个答案来自哪份文档的哪一段"。RAG 天然支持引用溯源，而纯生成模型难以给出可信来源。

下表汇总了 LLM 的主要痛点及 RAG 的应对方式：

| LLM 痛点 | 表现 | RAG 的应对 |
|---|---|---|
| 知识截断 | 不知训练后发生的事 | 知识库可实时更新 |
| 幻觉 | 编造看似合理的事实 | 提供权威依据降低编造 |
| 私有数据缺失 | 不了解企业内部知识 | 外挂私有知识库 |
| 更新成本高 | 重训昂贵且周期长 | 仅更新索引即可 |
| 不可溯源 | 答案缺来源 | 支持引用与追溯 |

## 三、RAG 与传统生成式 LLM 的对比

传统生成式 LLM 把所有"知识"压缩进模型参数里，回答时只依赖参数化的内部记忆。RAG 则把知识从参数中剥离出来，存放在外部的、可检索的知识库中，参数里只保留"语言能力"和"利用上下文作答"的能力。两者在多个维度上存在显著差异：

| 维度 | 传统生成式 LLM | RAG |
|---|---|---|
| 知识存储位置 | 模型参数（内部记忆） | 外部知识库（参数外） |
| 知识更新方式 | 重新训练 / 微调 | 更新文档与索引 |
| 更新成本 | 高（算力+数据+时间） | 低（近实时） |
| 事实准确性 | 易幻觉 | 有依据，更准确 |
| 私有数据支持 | 需训练进模型 | 直接接入知识库 |
| 可溯源性 | 弱 | 强（可引用来源） |
| 推理成本 | 单次前向传播 | 检索 + 生成两阶段 |
| 延迟 | 低 | 略高（多了检索步骤） |

需要强调的是，RAG 与传统生成并非对立关系，而是互补：RAG 仍然依赖一个强大的 LLM 作为"生成器"，它解决的是"知识从哪来、是否最新、是否可信"的问题，而不是"如何组织语言"的问题。

## 四、RAG 与知识库、搜索引擎的关系

RAG 并不是凭空出现的新事物，它站在两个成熟体系之上：**搜索引擎/检索系统**和**知识库管理**。

- **与搜索引擎的关系**：RAG 的检索环节本质上是一个"语义搜索引擎"。早期搜索引擎基于关键词匹配（如 BM25、TF-IDF），而现代 RAG 通常使用稠密向量检索（embedding + 近似最近邻），能够理解语义相似而不仅是字面重叠。可以说，RAG 把搜索引擎的检索能力与大模型的生成能力做了串联。
- **与知识库的关系**：RAG 中的"知识库"可以是企业文档库、FAQ、产品手册、代码库、数据库等。RAG 给静态知识库加上了一个"自然语言问答"的智能入口，让用户不必自己翻文档，而是由系统检索并总结。
- **与传统问答系统/专家系统**：相比早期基于规则或知识图谱的专家系统，RAG 用大模型的自然语言理解能力替代了僵硬的规则匹配，能处理更灵活多变的问题。

一句话概括：RAG = 检索引擎的"找" + 知识库的"存" + 大模型的"读与答"。

## 五、RAG 的核心三步：检索-增强-生成

RAG 的工作流程可以拆解为三个核心步骤，对应其名称中的三个单词。

### 1. 检索（Retrieve）

给定用户问题，系统从知识库中检索出最相关的若干文本片段。这一步通常包含：

- **文档切分（Chunking）**：把长文档切成合适大小的片段，便于检索与注入。
- **向量化（Embedding）**：用嵌入模型把问题和片段映射为高维向量。
- **相似度匹配**：在向量索引中用最近邻算法找出与问题向量最接近的片段。
- **可选重排（Rerank）**：用一个更精细的模型对初步结果重新排序，提升相关性。

检索质量直接决定 RAG 的上限——如果检索不到正确片段，再强的生成模型也无米下锅。

### 2. 增强（Augment）

把检索到的片段作为上下文，与用户的原始问题拼接成一个新的提示词（Prompt）。一个典型的增强后提示结构是：

> 请根据以下参考资料回答问题。若资料中无相关信息，请说明无法回答。
>
> 参考资料：[片段1] [片段2] [片段3]
>
> 问题：用户的问题

"增强"的精髓在于：模型不再是凭记忆作答，而是被要求"基于给定资料"作答。这不仅提供了事实依据，还通过提示约束降低了幻觉，并使答案可溯源。

### 3. 生成（Generate）

增强后的提示被送入 LLM，由它综合上下文与问题，生成自然语言答案。生成阶段负责语言组织、逻辑推理、多片段整合和风格控制。最终答案可以附带来源引用，形成完整的可追溯回答。

整个三步流程可以概括为：**找资料 → 喂资料 → 写答案**。

## 六、典型应用场景

RAG 在以下场景中表现尤为突出：

- **企业知识问答**：基于内部文档库构建员工/客户智能助手，回答流程、政策、产品问题。
- **客服与支持**：结合 FAQ 和历史工单，自动给出准确答复并附引用。
- **法律与医疗咨询**：在专业领域提供有据可查的辅助问答，降低幻觉风险。
- **代码与文档助手**：基于代码库或技术文档回答开发问题（如各类 Coding Copilot）。
- **实时信息问答**：接入新闻、行情、天气等实时数据源，回答时效性问题。
- **教育与研究辅助**：基于教材和论文库，提供可溯源的学习问答。

这些场景的共同点是：需要准确、最新、可溯源的领域知识，而这正是 LLM 的短板、RAG 的强项。

## 七、RAG 的局限性

RAG 并非银弹，它也有明显局限：

- **检索质量是瓶颈**：检索不到正确片段，答案必然出错；这是 RAG 失败的最常见原因。
- **延迟与成本增加**：多了一次检索（甚至重排）步骤，推理延迟和算力成本上升。
- **上下文长度限制**：能注入的片段数量受 LLM 上下文窗口限制，过长会丢失信息或增加成本。
- **复杂推理能力有限**：对需要多步推理、跨文档综合、数值计算的问题，简单 RAG 表现不佳，需进阶架构。
- **知识库构建与维护成本**：切分策略、索引更新、质量评估都需要工程投入。
- **并非消除幻觉**：RAG 降低了幻觉，但当检索片段相互矛盾或模型忽略上下文时，仍可能出错。
- **冷启动与召回评估难**：缺乏标注数据时，难以量化检索与整体效果。

## 小结

RAG 通过"检索-增强-生成"三步，把外部知识库与大模型的生成能力优雅地结合在一起，针对性地缓解了 LLM 的知识截断、幻觉、私有数据缺失和更新成本高等痛点。它不是对传统生成的颠覆，而是对其的增强与补充。理解 RAG 的动机与原理，是构建任何严肃大模型应用的第一步；而后续如何在检索质量、上下文管理和评估体系上持续优化，则是从"能用"走向"好用"的关键。
