# 稀疏检索与 BM25：词频、IDF 与 Okapi BM25 公式实践

在 RAG（检索增强生成）系统中，召回阶段决定了最终生成质量的天花板。稠密向量检索擅长语义相似度匹配，而稀疏检索（sparse retrieval）则承担着精确匹配、专有名词、数字与代码标识符等「字面级」命中任务。BM25（Okapi BM25）是稀疏检索事实上的工业标准，至今仍是 Elasticsearch、Lucene、OpenSearch 等搜索引擎的默认打分函数。本文从倒排索引出发，回顾 TF-IDF 的不足，推导 Okapi BM25 公式，逐项解释 IDF、词频饱和与文档长度归一化，并说明其在中文分词、稀疏检索优劣以及混合检索中的实践定位。

## 一、倒排索引基础

稀疏检索的高效性建立在倒排索引（inverted index）之上。与正向索引「文档 → 词项列表」不同，倒排索引建立的是「词项 → 文档列表」的映射，使检索时只需取出查询词对应的倒排表做交集即可，无需扫描全量文档。

一个倒排索引通常包含：

- **词典（dictionary）**：所有出现过的词项（term），通常用 FST（Finite State Transducer）或哈希表存储。
- **倒排表（postings list）**：每个词项对应的出现该词的文档 ID 列表，附带词频、位置、偏移等统计信息。
- **文档元数据**：文档长度、字段信息等，用于打分归一化。

构建流程为：文本 → 分词 → 词项归一化（小写、去停用词、词干提取）→ 更新词典与倒排表。查询时对查询串做同样的分词与归一化，再在倒排表上求交并打分。

倒排索引的核心优势是**查询复杂度与文档总数解耦**——它只遍历包含查询词项的文档，而非全库扫描。这也是稀疏检索在大规模语料上仍具竞争力的根本原因。

## 二、TF-IDF 回顾与不足

TF-IDF（Term Frequency–Inverse Document Frequency）是 BM25 的直接前身。其核心思想是：一个词对文档的区分度，既取决于它在文档中出现的频率（TF），也取决于它在整个语料中的稀有程度（IDF）。

经典 TF-IDF 打分（一种常见变体）：

```
score(q, d) = Σ_{t∈q} tf(t, d) · idf(t)

tf(t, d)   = 词项 t 在文档 d 中的出现次数
idf(t)     = log(N / df(t))     （N 为文档总数，df(t) 为包含 t 的文档数）
```

TF-IDF 直观但存在几个关键缺陷，这正是 BM25 要解决的问题：

1. **词频线性增长无上界**。一个词在文档中出现 100 次，得分是出现 1 次的 100 倍。实际上，词频的边际收益是递减的——出现第 10 次 vs 第 1 次差异显著，但第 100 次 vs 第 90 次几乎无信息增益。线性 TF 会让长文档或关键词堆砌的文档被过度打分。
2. **缺乏文档长度归一化**。长文档天然更容易匹配到查询词，且词频绝对值更高，导致长文档系统性地被高估。TF-IDF 没有机制纠正这种偏差。
3. **IDF 可能为负或为零**。当某词出现在超过半数文档时，`log(N/df)` 会变负甚至趋于零，破坏打分单调性。BM25 用 `log(1 + ...)` 形式规避了这一点。

## 三、Okapi BM25 公式与参数

Okapi BM25 由 Robertson 等人在 1990 年代提出，是概率检索模型（PRP）的工程化实现。它针对 TF-IDF 的上述缺陷引入了词频饱和函数和文档长度归一化。

### 3.1 完整公式

给定查询 q 与文档 d，BM25 打分：

```
score(q, d) = Σ_{t∈q} IDF(t) · TF_factor(t, d)

IDF(t)        = log(1 + (N - df(t) + 0.5) / (df(t) + 0.5))

TF_factor(t, d) = (tf(t, d) · (k1 + 1)) / (tf(t, d) + k1 · (1 - b + b · |d| / avgdl))
```

其中：
- `N`：语料中文档总数
- `df(t)`：包含词项 t 的文档数
- `tf(t, d)`：词项 t 在文档 d 中的出现频次
- `|d|`：文档 d 的长度（词项数）
- `avgdl`：语料平均文档长度
- `k1`：词频饱和参数，典型范围 `[1.2, 2.0]`，默认常取 `1.5`
- `b`：文档长度归一化参数，范围 `[0, 1]`，默认 `0.75`

### 3.2 IDF 逐项解释

BM25 的 IDF 采用 Robertson-Sparck Jones 形式：

```
IDF(t) = log(1 + (N - df + 0.5) / (df + 0.5))
```

- 分子 `(N - df + 0.5)` 表示「不包含 t 的文档数（加平滑）」，分母 `(df + 0.5)` 表示「包含 t 的文档数（加平滑）」。比值衡量 t 区分文档的能力。
- `+0.5` 是拉普拉斯平滑，避免 df=0 或 df=N 时出现极端值。
- 外层 `log(1 + ...)` 保证 IDF 始终非负：当某词出现在所有文档（df=N）时，IDF 退化为 `log(1 + 0.5/(N+0.5))` 而非负数，解决了 TF-IDF 的负 IDF 问题。

### 3.3 TF 饱和：k1 的作用

TF 因子是 BM25 相对 TF-IDF 最关键的改进：

```
TF_factor = (tf · (k1 + 1)) / (tf + k1 · (...))
```

分子中的 `(k1 + 1)` 是线性放大因子；分母中的 `tf` 使整体随 tf 增长但增速递减。当 `tf → ∞` 时，TF_factor 渐近上界为 `(k1 + 1)`，即词频贡献被「封顶」。

- **k1 控制饱和速度**。k1 越大，饱和越慢，词频作用越接近线性；k1 越小，饱和越快，第 2 次出现与第 10 次出现的差异被迅速抹平。
- k1=0 时 TF_factor 恒为 1，完全忽略词频，退化为布尔检索。
- 典型取值 1.2–2.0。语料中查询词重复度低（如查询多为短关键词）时 k1 可偏小；反之偏大。

### 3.4 文档长度归一化：b 的作用

分母中的 `(1 - b + b · |d| / avgdl)` 是长度归一项：

- `|d| / avgdl` 是文档相对平均长度。长文档该值 >1，短文档 <1。
- `b` 控制长度归一化的强度：
  - **b=1**：完全归一化，打分按文档长度严格缩放，长文档词频被充分折扣。
  - **b=0**：不做任何长度归一化，退化为不考虑长度的词频饱和模型。
  - **b=0.75**（默认）：经验上在多数语料上表现最佳，对长文档做部分而非完全惩罚，避免过度打压真正信息密集的长文档。

直觉理解：长文档匹配到查询词，可能只是因为「文档长所以碰巧出现」，未必更相关；b 项就是对这种「长度带来的虚假相关」做折扣，但保留一定弹性——因为有些长文档确实更全面。

## 四、BM25 与 TF-IDF 对比

| 维度 | TF-IDF | Okapi BM25 |
|---|---|---|
| 词频处理 | 线性增长，无上界 | 饱和函数，渐近 `(k1+1)` 上界 |
| 文档长度归一 | 无 | 有，由 b 参数控制强度 |
| IDF 形式 | `log(N/df)`，可能为负 | `log(1 + (N-df+0.5)/(df+0.5))`，恒非负 |
| 平滑 | 无 | `+0.5` 拉普拉斯平滑 |
| 理论基础 | 启发式 | 概率检索模型（PRP） |
| 参数 | 无 | k1、b 可调 |
| 长文档偏向 | 系统性高估长文档 | 通过 b 校正 |
| 工业地位 | 教学经典 | 搜索引擎默认（Lucene/ES） |

## 五、中文分词对 BM25 的影响

BM25 是「词项级」打分，分词质量直接决定召回上限。对中文等不含空格的语言，分词策略的影响尤为显著。

中文常见分词策略及其对 BM25 的影响：

- **词典分词（jieba/HanLP）**：切出语义完整的词，词项质量高，但依赖词典覆盖，未登录词（OOV）会被切碎或粘连。
- **字粒度（单字 unigram）**：召回率高、无 OOV，但词项歧义大（「银行」拆为「银」「行」后「行」歧义严重），IDF 区分度下降。
- **双字 bigram**：兼顾召回与区分度，是简单稳健的折中方案——本项目即采用此策略。
- **子词（BPE/WordPiece）**：与稠密检索的 tokenizer 对齐，但对 BM25 收益有限。

**关键结论**：中文分词的粒度需要在召回率与精确率间权衡。粒度过细导致 IDF 失真（高频单字稀释区分度），粒度过粗导致漏召（词典未覆盖的实体无法命中）。本项目采用「CJK 单字 + 相邻字 bigram」的混合粒度（见下节），是工程上务实的选择。

## 六、稀疏检索的优势与局限

### 6.1 优势

- **精确匹配**：对查询词的字面出现敏感，稠密检索的「语义相近但字面不同」无法替代字面精确命中。
- **专有名词与实体**：人名、产品型号、API 名、错误码等低频高区分度词项，BM25 的高 IDF 天然加权。
- **数字与代码标识符**：版本号、ID、函数名等无语义但需精确匹配的场景，稠密向量难以可靠处理。
- **可解释性**：打分可拆解为各词项的 IDF·TF 贡献，便于调试与归因。
- **无需训练**：零样本、无嵌入模型依赖，冷启动友好；索引构建成本低。

### 6.2 局限

- **无语义理解**：「手机」与「移动设备」字面不同则互不召回，同义词、近义词召回弱。
- **词形变化敏感**：英文 run/running/ran 若未做词干归一会被视为不同词项。
- **依赖分词质量**：中文分词错误会直接损害召回。
- **长查询稀释**：查询词越多，IDF 加权后噪声词项越容易干扰排序，需配合查询扩展或裁剪。

## 七、BM25 与稠密检索的互补关系

稠密检索将查询与文档编码为连续向量，用相似度（如余弦、点积）衡量语义相关性，擅长「意思相近但字面不同」的召回；BM25 擅长「字面精确匹配」。两者互补，构成混合检索（hybrid retrieval）：

- 稠密检索覆盖语义同义、改写、跨语言召回。
- BM25 覆盖专有名词、数字、代码、低频词精确命中。
- 融合方式常见有：RRF（Reciprocal Rank Fusion，倒数排名融合）、加权线性融合（归一化后加权求和）、级联（先 BM25 召回候选再向量精排）。

工程经验：纯稠密在「问实体名/编号」类查询上易翻车，纯 BM25 在「换种说法问同一件事」上易漏召，二者融合的鲁棒性显著优于单一方法。

## 八、本项目对照：Bm25ChunkSearcher

本项目的 `storage-layer` 模块实现了 `Bm25ChunkSearcher`，作为混合检索的稀疏分支，与基于 pgvector 的稠密向量检索组合成 hybrid retrieval。

### 8.1 角色定位

`VectorIndexService` 支持三种检索模式：`vector`、`bm25`、`hybrid`（默认）。hybrid 模式下，查询经 `QueryExpander` 扩展后，同时送往向量检索与 BM25 检索，各自取 `topK * 3` 候选，再通过排名融合（RRF 风格）合并去重，最终截断返回 topK。BM25 分支保证了对专有名词、代码标识符、版本号等字面查询的命中率，弥补稠密检索在这类「语义无关但需精确」查询上的短板。

### 8.2 打分实现要点

`Bm25ChunkSearcher` 采用标准 Okapi BM25 公式，参数取 `k1 = 1.5`、`b = 0.75`（工业默认值）。打分流程：

1. **分词**：`tokenize` 对文本做 NFKC 归一化与小写化后切分。对 ASCII 字母数字连续段作为一个拉丁词项；对 CJK（汉字/平假名/片假名/谚文）采用「单字 + 相邻二字 bigram」混合粒度——即每个汉字先作为 unigram 入索引，再与其相邻字组成 bigram 一并入索引。这种 bigram 策略在无依赖中文分词库的情况下，兼顾召回率与区分度。
2. **统计量计算**：遍历知识库下所有 chunk，计算每个 chunk 的 token 列表、语料平均长度 `avgdl`、各词项的文档频率 `df`，以及文档总数 `N`。
3. **打分**：对查询 token 去重后逐项累加。每项 IDF 取 `log(1 + (N - df + 0.5) / (df + 0.5))`，TF 因子为 `(tf · (k1+1)) / (tf + k1 · (1 - b + b · |d| / avgdl))`，与标准公式一致。
4. **过滤与排序**：丢弃得分为 0 的 chunk，按分数降序取 topK。

需注意，该实现每次查询都重算全量统计量（df、avgdl），适合中小规模知识库；若语料规模增长，应将倒排索引与统计量持久化（如落库或 Lucene），避免逐次全量扫描。

## 九、小结

BM25 是稀疏检索的基石：它用概率检索模型修正了 TF-IDF 的词频线性增长与缺乏长度归一两大缺陷，以 `k1` 控制词频饱和速度、`b` 控制文档长度归一强度，公式简洁、参数语义清晰、无需训练。其优势在于精确匹配、专有名词与数字命中、强可解释性与零样本冷启动；局限在于无语义理解、同义词召回弱、依赖分词质量。在 RAG 系统中，BM25 与稠密向量检索互补：前者保字面精度，后者拓语义广度，二者经排名融合构成鲁棒的混合召回。本项目的 `Bm25ChunkSearcher` 以标准 Okapi BM25（k1=1.5, b=0.75）与 CJK bigram 分词落地这一理念，与 pgvector 稠密检索协同，是混合检索的典型工程实践。
