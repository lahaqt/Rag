# 意图识别与多路由调度：路由到知识库、工具与实时搜索

## 一、为什么需要意图识别与路由

一个常见误区是把 RAG 等价于"检索—生成"单管道：凡提问一律走向量库检索，再把片段塞进提示词交给 LLM。这在真实场景会暴露三类问题。其一，**并非所有问题都需要检索**——闲聊强行检索既浪费算力又引入无关片段。其二，**并非所有问题都能靠静态知识库回答**——"今天天气""最新股价"的答案存在于知识库建成之后的外部世界，库里没有对应文档，硬答只会产生幻觉。其三，**有些问题需要动作而非信息**——"查我的订单""取消订单"期望调用业务接口完成状态变更，必须路由到工具。

意图识别与多路由调度的本质，是在"用户输入"与"最合适通道"之间建立决策层，回答：**这条 query 该交给谁**——直接回复、知识库检索、联网搜索，还是调用工具。好的路由层能同时获得准确性（实时问题不再胡编）、效率（闲聊不触发昂贵检索）、可扩展性（新增能力只需新增分支）。

## 二、意图分类体系

分类过粗路由无法区分，过细分类器准确率塌方。工程上较均衡的是五分类：

| 意图 | 典型查询 | 期望通道 | 是否检索 | 是否联网 |
|------|----------|----------|----------|----------|
| 闲聊 (chitchat) | "你好""谢谢" | 直接回复 | 否 | 否 |
| 知识库问答 (knowledge) | "年假怎么请""差旅标准" | 向量检索+生成 | 是 | 否 |
| 实时信息 (realtime) | "今天天气""最新股价" | web_search+生成 | 否 | 是 |
| 工具调用 (tool) | "查订单""取消订单" | 业务接口/函数调用 | 否 | 视接口 |
| 任务执行 (task) | "把这周日报汇总成周报" | 多步编排(ReAct) | 可能 | 可能 |

要点有四。其一，**闲聊与知识库边界**靠长度与关键词双判——短句命中寒暄词表归闲聊，否则默认检索，避免把"收到"误送向量库。其二，**实时信息最难单靠文本判别**，依赖隐含的"时效性"：同一"产品价格"在知识库更新当天可检索回答，一周后应转实时搜索，故判定要结合时间词（"今天/最新/现在"）与领域词（"天气/股价/新闻"）。其三，**工具与任务区分粒度**：工具调用是单次函数调用可完成的事务，任务执行是需规划—执行—观察多步循环的复杂目标，更适合 ReAct 编排。其四，**澄清意图作兜底类**：查询过短或歧义且无上下文时（孤立的"怎么办"），与其乱猜不如反问。

## 三、路由决策依据与策略

工程上有三种策略，常组合使用。**规则路由**用关键词/正则/长度阈值直接判定，确定、零成本、可解释，但覆盖面有限，适合高频固定边界（实时词表、闲聊精确匹配）。**分类器路由**训练轻量模型（fastText、BERT 微调、句向量+KNN）输出意图与置信度，能捕捉语义、对未见表达有泛化，但需标注数据且有推理开销。**LLM 路由**把意图判定交给 LLM 经结构化输出返回，零数据冷启动、对复杂模糊查询理解力最强，但延迟高、成本高、输出需 schema 约束。

落地推荐**分层级联**：规则快速短路→未命中交分类器或 LLM 语义判定→兜底规则（默认知识检索）保底。决策依据除意图标签外，还应纳入**置信度**（低于阈值走 fallback 或追问）、**知识库是否指定**（未指定时即便意图为 knowledge 也无法检索，应转直接回复）、**历史上下文**（有无会话直接影响短查询是否歧义）。

## 四、实时信息判断：是否需要 web_search

实时分支最易出错也最关键，因为答错的方式不是"无结果"而是"用陈旧知识库内容一本正经地胡编"。可从三个信号综合决策。**时间信号**：query 含显式时间词（"今天/现在/最新/昨天"）是最强单信号，规则即可捕获。**领域信号**：query 落在强时效领域（天气、新闻、股价、汇率、赛事），几乎天然意味着"现在"的答案，即便用户没说"今天"。**知识库时效信号**：即便无时间词，若命中高频变动主题且库更新滞后，也应联网——此信号较难精确判断，工程上常以前两信号为主，必要时结合"检索置信度低→回退联网"的二次决策。

需特别处理**查询增强**：实时分支确定后，原始 query 常需补全才有有效结果。如"今天天气怎么样"直搜可能指向任意城市，应清洗"怎么样/呢/吗"等口语填充词并补"预报"关键词，必要时注入当前日期。

## 五、工具调用路由

工具调用意图对应"执行动作"，路由目标是业务接口。相比实时搜索多了两个维度。**工具选择**——从已注册工具集（订单查询、物流、地址修改、退货）中选出匹配者，可用工具元数据语义匹配或交 LLM function calling 自动选择。**参数抽取**——"查订单号 X123 的物流"需抽出 `order_id=X123`，依赖 LLM 结构化输出或槽位填充；参数不全应路由到"追问补全"而非盲目调用。工具结果还应携带**回退策略**：调用失败（超时、权限不足、参数非法）时降级为失败提示或回退知识库检索相近流程说明，并把 `finishReason` 标记为 `web_search_failed` 类可观测信号便于排查。

## 六、多路由并行与 fallback

复杂查询常非单一路由可解。**并行调度**适用于"答案来自多个独立源"：如"某产品最新功能及官方文档说明"——最新功能需联网、文档说明需检索，无依赖可并行触发，最后由生成层融合；代价是资源翻倍，需注意结果去重与冲突消解。**级联 fallback**则是"主路径失败再走备选"：实时搜索失败→降级知识库；知识库无结果→回退联网；检索+生成都置信不足→转澄清追问。设计要点是**快速失败**（设超时阈值）与**可观测**（记录每路命中与否及原因）。

典型级联路由决策链路（规则短路→意图判定→通道选择→fallback 兜底）：

```
            用户 query
                │
     ┌──────────▼──────────┐
     │  规则短路(关键词/长度) │
     └──────────┬──────────┘
   命中实时词? 命中闲聊词? 命中工具词?
        │          │          │
        ▼          ▼          ▼
    web_search  直接回复   工具调用
        │          │          │
        └────┬─────┘          │
             ▼                │
     意图分类器/LLM 路由       │
     (intent+confidence)      │
             │                │
    ┌────┬───┴───┬────────┐   │
    ▼    ▼       ▼        ▼   │

    │    │       │        │   │
    ▼    ▼       ▼        ▼   │

    │    │       │

```

## 七、路由失败与兜底回答

路由层失败比生成层更隐蔽，典型有三类。**分类器置信度过低**——多类间摇摆，兜底为设阈值，低于阈值走最安全通道（默认知识检索）并降级，或转澄清追问。**路由服务不可用**——下游（如 query-rewrite-service）超时或宕机，兜底为返回预设默认分析结果（意图 knowledge、置信度 0.5、路由 knowledge_retrieval），保证主链路不中断并打 warn 日志。**通道内失败**——已正确路由但执行失败（web_search 异常、工具报错、库无结果），兜底分两层：失败给可解释错误提示，无结果给"未找到依据"的诚实回答而非编造。

兜底统一原则是**诚实优先**：宁说"没找到依据/无法确认"也不硬答，要求 Prompt 强约束（"不能从片段推出的内容要明确说明没有依据"），并把失败原因以结构化字段透出，方便前端展示与运维定位。

## 八、路由与查询重写的协同

意图识别、路由与查询重写三者紧密耦合，协同不好会出现"意图判对、路由走对、但 query 没改写导致检索空"的尴尬。正确关系是：**意图识别先行，路由决策与改写并行或紧随，改写结果供通道消费**。意图为 knowledge 时改写把口语化、带指代的 query 改写为检索友好的规范 query（补全代词、去停用词、生成检索子 query）；为 realtime 时改写做实时 query 富化（清洗口语词、补时间与领域关键词）；为 tool 时改写甚至可兼任参数抽取。这意味着 query-rewrite-service 通常同时输出 `intent`、`route`、`rewrittenQuery` 与 `retrievalQueries`，路由层消费 intent/route 决定通道，通道消费改写结果执行检索或搜索——这种"一次分析、多处消费"避免了意图判定与改写各跑一遍 LLM 的重复开销。

## 九、本项目对照：意图识别与路由的分工

本项目将意图识别与路由拆分到两个微服务，形成"分析在前、路由在后、通道执行"的分工。

**query-rewrite-service** 负责意图识别与路由建议。其 `IntentClassifier` 采用规则分层：先用长度+精确匹配词表判定闲聊（"你好/谢谢/再见"短句命中即归 chitchat），再用工具关键词词表（"我的订单/查订单/物流状态/年假/取消订单"）判定工具意图，对无历史且过短或歧义的查询（"怎么办/帮我看看"）归澄清，其余默认知识检索。意图经 `QueryIntent` 枚举统一映射到路由标签——KNOWLEDGE→`knowledge_retrieval`、TOOL→`tool_invocation`、CHITCHAT→`direct_reply`、CLARIFICATION→`ask_clarification`，连同置信度、改写 query、检索子 query 一并经 `QueryAnalysisResponse` 返回，体现"一次分析、多处消费"。

**agent-service** 的 `ToolRouter` 是路由执行层，`decide()` 采用两级判定：第一级规则短路，query 小写化后扫描实时关键词词表（"今天/现在/最新/天气/股价/today/weather"等），命中即返回 `web_search` 决策并附带富化后 query；第二级消费分析结果，当 `intent=tool` 或 `route=tool_invocation` 时同样路由到 `web_search`（当前将工具意图统一映射到联网通道，未来可扩展为多工具分发）。`ToolDecision` 以 record 封装 `useTool/toolName/query/reason`，reason 字段（如 `contains_realtime_keyword:今天`）提供可观测的路由依据。

`ChatOrchestrator` 是协同枢纽：先调 `analyze()` 拿分析结果（含降级兜底——分析服务异常时回退为默认 knowledge 分析），再调 `ToolRouter.decide()`；若决策为 web_search 则走联网通道（`WebSearchTool` 检索+专用 Prompt 生成，失败时降级为失败提示草稿并标记 `web_search_failed`），否则按 `shouldRetrieve()` 判断——仅当指定知识库且意图/路由为 knowledge 时才检索，闲聊与澄清天然短路检索。这种"规则短路+分析结果+兜底降级"的分层结构，正是前述级联路由的工程落地。

## 小结

意图识别与多路由调度是 RAG Agent 从"单管道检索生成"走向"多通道智能调度"的关键跃迁。核心在于：先建立覆盖闲聊/知识库/实时/工具/任务的均衡意图体系，再用"规则短路+分类器/LLM 语义判定+默认兜底"的级联策略把意图映射到通道；实时信息靠时间词与领域词双信号判别并配 query 富化；工具调用需额外解决选择与参数抽取；多路由通过并行与 fallback 兼顾覆盖面与健壮性；路由失败以诚实兜底优先并把原因结构化透出。意图识别、路由决策与查询重写应"一次分析、多处消费"协同——本项目以 query-rewrite-service 输出 intent/route/rewrittenQuery、agent-service 的 ToolRouter 做规则短路与结果融合、ChatOrchestrator 统一编排与降级，正是这一理念的可参考落地。
